# 形式的導出の構成 (Γ ⊢ φ)

タスクを受け取り、Lean 4 を用いて Γ ⊢ φ の導出を構成する。

## 前提知識

このスキルは以下のファイルを参照する:

1. `docs/formal-derivation-procedure.md` — 手順書の原典
2. `docs/mathematical-logic-terminology.md` — 数理論理学用語リファレンス
3. `lean-formalization/Manifest/Terminology.lean` — 用語リファレンスの Lean 形式化
4. `lean-formalization/Manifest/Procedure.lean` — 手順書の規則の Lean 形式化
5. `lean-formalization/` — 既存の Lean 形式検証（参照用）

**デプロイ前提:** このスキルは agent-manifesto プロジェクト構造内での利用を前提とする。
上記ファイルおよび `.claude/agents/verifier.md`（Phase 4 独立レビューで使用）が
プロジェクト内に存在すること。

## 設計原則: axiom 最小化

**axiom の数を最小化することを常に目指す。**

実戦で得た知見: 多くのタスクで T₀ は型定義（定義的拡大）のみで
エンコード可能であり、axiom 0 個で φ を導出できる。

```
axiom 0 の導出 > axiom ありの導出
```

理由:
- axiom 0 = `propext` のみに依存 → 公理衛生検査が自明に合格
- axiom 0 = 反例探索（Phase 4c 第 3 層）が不要
- axiom 0 = 体系の無矛盾性について心配する必要がない

**axiom が必要になるのは**、型体系の外にある情報
（docstring の内容、外部ファイルの構造等）に依存する場合のみ。
この場合も opaque 述語 + axiom の使用を最小限に抑える。

## 役割分担

```
LLM（生成者）                    Lean コンパイラ（検証オラクル）
─────────────                    ──────────────────────────────
論議領域を型として定義する        型の整合性を検証する
目標命題 φ を定式化する           定義の well-formedness を検証する
前提集合 Γ を構築する             導出の正しさを検証する
Γ ⊢ φ の導出を試みる             エラーメッセージを返す
エラーを解釈し修正する            （判断しない）
戦略を選択・変更する
```

## 入力

ユーザーから以下を受け取る:

- **タスク記述**: 達成すべきことの自然言語記述
- **出力先**: Lean ファイルのパス（省略時はユーザーに確認）
- **制約**: タスクに付随する既知の制約・前提（任意）

## 実行手順

### Phase 1: Lean での構築

タスク記述から Γ ⊢ φ の構成要素を定義し、Lean ファイルとして構築する。

#### Step 1.1: 論議領域の定義

命題が語る対象を Lean の型として定義する。
型定義は Γ にも φ にも属さず、両者が共有する語彙を構成する。

**既存型の再利用を優先する。** Terminology.lean や Procedure.lean、
FormalDerivationSkill.lean に関連する型がすでに定義されている場合、
import して再利用する。

```lean
-- 例: SLA を扱うタスクの論議領域
structure Request where
  size : Nat
  priority : Priority

structure System where
  capacity : Nat
  cacheEnabled : Bool
```

#### Step 1.2: 目標命題 φ の定式化

タスクの達成条件を命題 φ として定式化する。

**φ の構成原則:**
- 一つのタスクに一つの目標命題。複合タスクは複数の命題に分解する
- 自然言語の意図を必ず docstring に残す（形式化ギャップ検証の基盤）
- 最初は `sorry` で全体構造を確立する

```lean
/-- [目標命題]
    タスク: 「...（自然言語の意図）...」
    形式化の意図: ...
-/
theorem goal_proposition :
  ∀ ..., ... :=
  sorry  -- Phase 2 で導出を構成する
```

#### Step 1.3: Γ / φ 境界の判定

タスクから得られた各命題 P について:
- P が「成立していることを前提とする事実」→ Γ に加える（前提）
- P が「成立させるべき条件」→ φ の一部とする（導出目標）

演繹定理により Γ, P ⊢ Q と Γ ⊢ P → Q は形式的には等価だが、
Γ に加えた命題は修正ループでの縮小対象となる点が実用上異なる。

**axiom 最小化の観点:** まず axiom なし（全て定義的拡大 + theorem）で
導出を試みる。導出不能な場合にのみ axiom を Γ に追加する。

#### Step 1.4: 前提集合 Γ の構築

Γ を基底理論 T₀ と拡大部分に分けて構築する。

**基底理論 T₀（修正ループで縮小しない）:**
- 契約由来 — 人間との合意、SLA、法的制約
- 自然科学由来 — 物理法則、数学的事実
- 環境由来 — ハードウェア制約、API 仕様

**T₀ のエンコード方法の選択:**
T₀ の主張が型定義（定義的拡大）で表現可能な場合、axiom ではなく
型定義 + theorem で表現する。列挙型の網羅性のように CIC から
導出可能な性質を axiom にすると、公理衛生検査 2（非論理的妥当性）に
違反する。T₀ の権威は型の構成子の選択に反映される。

- 型定義で表現可能 → 型定義（定義的拡大）+ theorem で証明
- 型定義で表現不能 → axiom（公理カード必須）

Procedure.lean が T₀ エンコード方法の安全性を形式的に証明済み:
`definitional_encoding_safer`: 型定義は axiom より厳密に安全。

**拡大部分 Γ \ T₀（修正ループで縮小・修正可能）:**
- 仮説由来 — LLM が推論から設定した仮説。反証可能
- 設計由来 — ドメインモデルの前提。設計判断に基づく

#### Step 1.5: T₀ の無矛盾性簡易検査

T₀ の公理を構築した直後に、明らかな矛盾の早期検出を行う。

**検査内容:**
- 互いに否定を主張する公理の組がないか
- 同じ項に対して矛盾する値を主張する公理がないか
- `lake build` で T₀ のみのコンパイルが通るか

**注意:** 公理系の無矛盾性を完全に検証することは一般に不可能である
（ゲーデルの第二不完全性定理）。この検査の目的は明白な矛盾の早期検出であり、
無矛盾性の証明ではない。

**axiom 0 の場合:** T₀ が型定義のみで構成されていれば、
定義的拡大は常に保存拡大であるため、この検査は自明に合格する。

#### Step 1.6: 公理カードの記載

Γ の各公理には以下のメタデータを Lean の docstring として記載する。

```lean
/-- [公理カード]
    所属: T₀（契約由来） or Γ \ T₀（仮説由来）
    内容: 公理が何を主張するか
    根拠: なぜこの公理を仮定するか
    ソース: 根拠の出典
    反証条件: （Γ \ T₀ の場合のみ必須）どのような観察で撤回するか
-/
axiom axiom_name : ...
```

| フィールド | T₀ の公理 | Γ \ T₀ の公理 |
|-----------|----------|--------------|
| **所属** | 必須 | 必須 |
| **内容** | 必須 | 必須 |
| **根拠** | 必須 | 必須 |
| **ソース** | 必須 | 必須 |
| **反証条件** | — | 必須 |

Procedure.lean が公理カード規則を形式的に定義済み:
`fieldRequired`: 区分とフィールドの要否を計算する関数。
`refutation_cond_rule`: 反証条件は Γ \ T₀ のみ必須。

#### Step 1.7: Lean コンパイル検証

各ステップで `lake build` を実行し、型エラーを解消する。

```bash
export PATH="$HOME/.elan/bin:$PATH" && lake build <target>
```

この段階でのエラーは主に型の不整合（定義的拡大で対処）。

### Phase 2: Γ ⊢ φ の導出の構成

**入力:** コンパイルが通る Lean ファイル（φ に sorry が残存）
**出力:** sorry が解消された Lean ファイル

#### Step 2.1: φ の分解

目標命題 φ の導出に必要な補題 (lemma) を同定する。

#### Step 2.2: ボトムアップ導出

依存関係の少ない補題から順に導出を構成する。

**頻出する証明パターン（実戦知見）:**

```lean
-- 列挙型の網羅性
theorem exhaustive : ∀ (x : MyType), x = .a ∨ x = .b ∨ x = .c := by
  intro x; cases x <;> simp

-- 合取の構成
theorem conj : P ∧ Q ∧ R := by
  refine ⟨?_, ?_, ?_⟩ <;> rfl

-- 関数定義の等式
theorem fn_eq : myFn .someCase = expectedValue := by rfl

-- 不等式の証明（simp で解決）
theorem ineq : a.strength > b.strength := by
  simp [strength]
```

#### Step 2.3: φ の導出

補題を組み合わせて Γ ⊢ φ の導出を完成する。

各導出の構成後に `lake build` を実行する。
コンパイルが通らない場合は Phase 3（修正ループ）に入る。

### Phase 3: 修正ループ

**入力:** Lean コンパイラのエラーメッセージ
**出力:** 修正された Lean ファイル、または戦略変更の判断

#### 3a: エラー解釈と修正アクション

| エラー種別 | 修正アクション | 修正の分類 |
|-----------|--------------|-----------|
| `type mismatch` | 論議領域の型定義を修正 | 定義的拡大 |
| `unknown identifier` | 定義の追加 or import 追加 | 定義的拡大 |
| `Invalid field 'foo'` | 別名前空間の型にドット表記不可。スタンドアロン関数に変更 | 定義的拡大 |
| `tactic 'simp' failed` | 別のタクティクを試行、補題を追加 | — |
| `tactic 'rfl' failed` on `∧` | `refine ⟨?_, ?_⟩ <;> rfl` に変更 | — |
| `unsolved goals` | 追加の導出ステップ、補題分解 | — |
| `declaration uses 'sorry'` | sorry の解消 or Γ への公理追加検討 | 公理追加は拡大 |

**Lean 4 固有の注意点（実戦知見）:**

- **名前空間とドット表記:** `namespace A` 内で定義した関数 `B.foo` は、
  型 `B` が別の名前空間に属する場合にドット表記 `x.foo` で呼べない。
  スタンドアロン関数 `fooOfB` を使う
- **`open` の活用:** `open Manifest.FormalDerivationSkill` で
  他モジュールの型を修飾なしで使える
- **`cases <;> simp` パターン:** 列挙型の網羅性証明に最も頻出

#### 3b: 修正の規律

修正が体系の拡張に該当する場合、以下の分類を適用する:

**定義的拡大（常に安全）:**
- 新しい型・関数・述語の定義追加
- 既存定義の型修正（コンパイルが通る範囲で）

**Γ \ T₀ の追加・変更（無矛盾な拡大であることを確認）:**
- 新しい公理の Γ への追加 → T₀ との矛盾チェック
- 既存の拡大公理の修正 → 依存する定理の再導出
- 反証条件付きの公理カードを必ず付与

**T₀ の縮小（禁止）:**
- T₀ の公理に対する縮小が必要と判断した場合 → ワークフローを終了（§4.3 未決）
- T₀ のどの公理が導出を阻害しているかを特定して報告

Procedure.lean が T₀ 縮小禁止を形式的に証明済み:
`t0_contraction_forbidden`: `permittedOp .baseTheory .contraction = false`

**φ の弱化（形式化ギャップの再検証必須）:**
- 目標命題 φ の条件を緩和する場合
- Step 1.3 の Γ / φ 境界判定を再実行する
- 弱化後の φ が元のタスクの意図をまだ捉えているか、Phase 4 で検証

#### 3c: 終了判定

**成功:** 全 `sorry` が解消され、`lake build` 成功

**戦略変更（以下のパターンを検出した場合）:**
- **同一エラー反復:** 同じエラーが 3 回連続で出現
- **公理膨張:** Γ \ T₀ の公理数が修正ループ開始時の 2 倍を超過
- **複雑度増大:** 導出のステップ数が単調に増加し続ける

戦略変更の選択肢（優先順位順）:
1. Γ \ T₀ の見直し（異なる前提を試す）
2. 論議領域の再定義（異なるモデル化を試す）
3. φ の分解方法を変更する
4. φ の弱化（最終手段。形式化ギャップの再検証必須）

Procedure.lean が優先順位を形式的に定義済み:
`revise_extension_first` / `weaken_goal_last`

**バックトラック時の保持/破棄ルール:**
- T₀: 常に保持。戦略変更で縮小しない
- Γ \ T₀: 戦略変更時に縮小・破棄可能
- 論議領域: 戦略 2 の場合のみ破棄
- 導出済み補題: 依存する公理・定義が変更されない限り保持

Procedure.lean が保持規則を形式的に定義済み:
`t0_always_retained` / `extension_most_disposable`

**タイムアウト:** 修正試行の上限回数到達、全戦略網羅で不成功

**Γ ⊬ φ の判定:**
- Γ ⊢ ¬φ が導出可能（反例の構成的証拠）
- Γ が無矛盾でない（Γ ⊢ ⊥ が導出可能）

### Phase 4: 監査

**入力:** Γ ⊢ φ の導出が完成した Lean ファイル（sorry なし）
**出力:** 監査報告

#### 4a: 導出の完全性確認

- [ ] `lake build` で警告なしにコンパイル成功
- [ ] 各定理に対して `#print axioms` を実行し、依存する公理を列挙
- [ ] Lean 基盤公理（`Classical.em`, `Classical.choice`, `propext`, `Quot.sound` 等）の使用を記録

**axiom 0 の場合:** `#print axioms` の出力が `[propext]` のみ、
または `does not depend on any axioms` であれば、
公理衛生検査（4b）は自明に合格する。

#### 4b: 公理衛生チェック（5 検査項目）

1. **非空虚性**: 各公理の前件が充足可能か（Empty 型や矛盾した条件がないか）
2. **非論理的妥当性**: 各公理が `by simp` や `by rfl` のみで証明可能でないか
3. **独立性**: 各公理を除去しても他の公理から導出不能か
4. **最小性**: 各公理を除去すると Γ ⊢ φ の導出が成立しないか
5. **基底理論の保存**: Γ は T₀ の無矛盾な拡大か

#### 4c: 形式化ギャップ検証（3 層）

**第 1 層: docstring 検査**
- φ の docstring と形式的定義の対応
- 各公理カードの「内容」フィールドと Lean の公理文の対応
- Γ / φ の境界判定が適切か

**第 2 層: 独立レビュー（P2 準拠）**
独立したコンテキストに以下を提示してレビューさせる:
- 元のタスク記述（自然言語）
- Γ ⊢ φ の導出
- Γ の公理一覧（公理カード付き）

レビュー観点:
- 形式化がタスクの意図から逸脱していないか
- Γ の公理が過度に強くないか（φ を自明にしていないか）
- 導出が空虚でないか（前件が非現実的でないか）

**形式化ギャップの具体例（実戦知見）:**

```
悪い例:
  def preservesConsistency : Op → Bool
    | .expansion => false    -- 「常に破壊する」と主張している
  文書の意図: 「破りうる」（may not preserve）

良い例:
  inductive ConsistencyGuarantee where
    | guaranteed      -- 定義により常に保存
    | notGuaranteed   -- 保存が保証されない（破壊するとは限らない）
  def consistencyGuarantee : Op → ConsistencyGuarantee
    | .expansion => .notGuaranteed
```

Bool の false/true で「ありうる/ありえない」を表現すると、
「常にそうである」と主張してしまう。程度や可能性を表す場合は
専用の型を定義する。

**独立レビューの実施方法:** `/verify` スキルを呼び出すか、
Verifier サブエージェント（`.claude/agents/verifier.md`）を直接起動する。

**第 3 層: 反例探索**
1. Γ をすべて満たすが、直観的にタスクが達成されないシナリオを構成する
2. そのようなシナリオが構成可能なら、形式化にギャップがある
3. ギャップが見つかった場合 → Γ の修正 or φ の修正 → Phase 1 に戻る

## 終了条件

### 成功: Γ ⊢ φ の導出完了

Phase 4 の監査を通過した状態。成果物:
- Γ ⊢ φ の導出（Lean ファイル）
- Γ の公理一覧（T₀ / Γ \ T₀ の分類、各公理カード付き）
- 導出の公理依存性（`#print axioms` の出力）
- 監査報告

### 失敗: Γ ⊬ φ の判定

- Γ ⊢ ¬φ が導出可能（φ の否定の構成的証拠）
- Γ が無矛盾でない（Γ ⊢ ⊥ が導出可能）

### 未決

修正ループの終了条件に到達し、Γ ⊢ φ も Γ ⊢ ¬φ も構成されていない状態。
導出済みの補題と残存する sorry の一覧を報告する。

## Lean 形式化との対応表

本スキルの概念は以下の Lean ファイルで形式的に定義・証明済み:

| スキルの概念 | Lean ファイル | 定理/定義 |
|------------|-------------|----------|
| T₀ 縮小禁止 | Procedure.lean | `t0_contraction_forbidden` |
| Γ \ T₀ 全操作許容 | Procedure.lean | `extension_all_ops_permitted` |
| 型定義 > axiom の安全性 | Procedure.lean | `definitional_encoding_safer` |
| 反証条件は Γ \ T₀ のみ | Procedure.lean | `refutation_cond_rule` |
| Phase 順序 | Procedure.lean | `phase_sequential` |
| 修正の安全性順序 | Procedure.lean | `modification_safety_chain` |
| T₀ は常に保持 | Procedure.lean | `t0_always_retained` |
| 拡大の包含関係 | Terminology.lean | `extension_strength_chain` |
| 定理の形式的同等性 | Terminology.lean | `theorem_roles_formally_equivalent` |
| AGM 操作の無矛盾性保証 | Terminology.lean | `agm_consistency_contrast` |

### 更新の互換性分類

- **conservative extension**: 例の追加、説明の明確化、Lean 対応表の更新
- **compatible change**: Phase 内の手順の微修正、出力フォーマットの軽微な変更
- **breaking change**: Phase の追加・削除・順序変更、公理衛生チェック項目の変更

### 更新の検証

更新後、以下を実施する:
1. 既知のタスクで導出を再実行し、手順が機能することを確認
2. 手順書（`docs/formal-derivation-procedure.md`）との整合性を確認
3. Lean 形式化（Terminology.lean, Procedure.lean）との整合性を確認
4. 破壊的変更の場合、過去の導出成果物への影響を評価する
