# Verifier (P2: 認知的役割分離)

Worker（あなた自身）の成果物を、独立したコンテキストで検証する。

## なぜ独立検証が必要か

E1（検証の独立性）: 同一プロセスによる生成と評価は構造的に信頼できない。
自分が書いたコードを自分でレビューしても、同じバイアスが働く。
独立したコンテキスト（Subagent）で検証することで、バイアスを分離する。

## 検証手順

1. **対象の特定**: ユーザーに検証対象を確認する（変更されたファイル、git diff 等）
2. **Subagent 起動**: 以下の条件で Verifier Subagent を起動する
   - `subagent_type: general-purpose`（またはプロジェクトの verifier agent）
   - Worker の中間状態（思考過程、仮説）を渡さない — 成果物のみ
   - 検証基準を明示する
3. **結果の報告**: Verifier の verdict をユーザーに報告する
4. **FAIL の場合**: 問題箇所と修正提案を提示し、ユーザーに判断を委ねる

## Subagent への指示テンプレート

```
あなたは独立した Verifier です。以下のファイルを検証してください。

検証対象: [ファイルパスのリスト]
検証基準:
- ロジックエラーがないか
- セキュリティ上の問題がないか
- テストが十分か
- L1（倫理・安全境界）に違反していないか

以下の形式で回答してください:
VERDICT: PASS または FAIL
ISSUES:
- (問題のリスト、なければ "none")
RECOMMENDATION: (推奨事項)
```

## 検証のスコープ判断

| 変更の種類 | 検証の深度 | 理由 |
|-----------|----------|------|
| L1 関連（安全・権限） | 必須 + 人間レビュー | D1: L1 は構造的強制 |
| 構造変更（アーキテクチャ） | 必須 | P3: 破壊的変更の可能性 |
| 通常のコード変更 | 推奨 | P2: 確率的出力の検証 |
| ドキュメント変更 | 任意 | リスクが低い |
