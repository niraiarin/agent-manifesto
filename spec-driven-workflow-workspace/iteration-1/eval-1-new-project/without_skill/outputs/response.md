# IoT デバイス監視システム — 仕様駆動で始めるためのガイド

## 質問への回答

IoT デバイスの監視システム（センサーデータの異常検知）を仕様駆動で始める場合、以下の順序で進めることを推奨します。

---

## Step 1: 要件の構造化（仕様の種を作る）

まず、曖昧な要件を構造化された仕様に変換します。

### 1a. ドメインモデルの定義

以下の概念を明確にします:

- **センサー**: どのような種類か（温度、湿度、振動、電流など）
- **データストリーム**: サンプリング間隔、データ形式、欠損の許容度
- **異常の定義**: 何をもって「異常」とするか（閾値超過、パターン逸脱、変化率異常など）
- **アクション**: 異常検知時に何をするか（アラート、シャットダウン、ログ記録）

### 1b. 制約の列挙

- レイテンシ要件（リアルタイム性がどこまで必要か）
- デバイス台数のスケール
- ネットワーク帯域・接続安定性
- データ保持期間

---

## Step 2: 仕様書の作成

### 2a. インターフェース仕様

まずシステムの境界を定義します:

```
入力: センサーデータストリーム (device_id, timestamp, metric_name, value)
出力: 異常イベント (device_id, timestamp, anomaly_type, severity, evidence)
```

### 2b. 異常検知ルールの形式化

異常検知ロジックを検証可能な形で記述します:

```
ルール1: value > threshold_high OR value < threshold_low → 閾値異常
ルール2: |value - moving_average(window=N)| > k * stddev → 統計的異常
ルール3: 連続 M 個のデータポイントが単調増加/減少 → トレンド異常
ルール4: データ到着間隔 > expected_interval * 2 → 欠損異常
```

各ルールに対して:
- 正常時の入力例と期待出力
- 異常時の入力例と期待出力
- 境界条件（ちょうど閾値の場合など）

を明記します。

### 2c. 非機能要件の定量化

- 検知遅延: センサーデータ到着から異常通知まで X 秒以内
- 偽陽性率: Y% 以下
- 偽陰性率: Z% 以下
- 可用性: 99.X%

---

## Step 3: テスト計画の導出（仕様からテストへ）

仕様駆動の核心は「仕様からテストを導出する」ことです。

### 3a. 受け入れテスト（Acceptance Tests）

仕様の各ルールに対応するテストケースを先に書きます:

```
テスト: 温度センサーが閾値 (80°C) を超えた場合、30秒以内にアラートが発生する
テスト: 正常範囲内のデータが1000件連続した場合、アラートは0件
テスト: センサーが60秒間データを送信しない場合、欠損異常が検知される
テスト: 移動平均から3σ以上逸脱した場合、統計的異常として検知される
```

### 3b. プロパティベーステスト

異常検知に特に有効:

- 「正常データのみを入力した場合、異常は検知されない」（偽陽性の排除）
- 「既知の異常パターンを入力した場合、必ず検知される」（偽陰性の排除）
- 「検知結果は入力順序に対して単調」（同じデータを再送しても結果が変わらない）

---

## Step 4: アーキテクチャの選定

テスト可能な仕様ができた時点で、アーキテクチャを決めます。

### 典型的な構成

```
[センサー群] → [データ収集層] → [ストリーム処理層] → [異常検知エンジン] → [通知層]
                     |                    |                     |
                     v                    v                     v
                [時系列DB]          [状態ストア]           [アラートDB]
```

### 技術選択の指針

| 層 | 選択肢の例 | 選定基準 |
|---|---|---|
| データ収集 | MQTT, HTTP, CoAP | デバイス制約、ネットワーク環境 |
| ストリーム処理 | Kafka Streams, Flink, 自前実装 | スケール要件、レイテンシ要件 |
| 異常検知 | ルールベース、統計モデル、ML | 精度要件、解釈可能性要件 |
| 時系列DB | InfluxDB, TimescaleDB, QuestDB | クエリパターン、データ量 |
| 通知 | Webhook, メール, Slack | 運用チームの要件 |

小規模なら全て自前実装でシンプルに始め、スケールが必要になった段階で分離するのが現実的です。

---

## Step 5: 最小実装（仕様に対するテストを通す）

### 推奨する最初の実装スコープ

1. **1種類のセンサー**（例: 温度）のデータを受け取る
2. **1種類の異常検知**（例: 閾値超過）を実装する
3. **受け入れテストを全て通す**

この最小スコープで仕様 → テスト → 実装 → 検証のサイクルを1周回します。

### ディレクトリ構成例

```
iot-monitor/
├── spec/                    # 仕様書
│   ├── domain-model.md
│   ├── anomaly-rules.md
│   └── non-functional.md
├── tests/
│   ├── acceptance/          # 受け入れテスト（仕様から導出）
│   ├── property/            # プロパティベーステスト
│   └── integration/         # 結合テスト
├── src/
│   ├── ingestion/           # データ収集
│   ├── detection/           # 異常検知エンジン
│   ├── storage/             # データ永続化
│   └── notification/        # 通知
└── deploy/                  # デプロイ設定
```

---

## まとめ: 仕様駆動の進め方

| 順序 | やること | 成果物 |
|---|---|---|
| 1 | ドメインモデル定義 | domain-model.md |
| 2 | 異常検知ルールの形式化 | anomaly-rules.md |
| 3 | 非機能要件の定量化 | non-functional.md |
| 4 | テストケースの導出 | tests/acceptance/ |
| 5 | 最小アーキテクチャ決定 | アーキテクチャ図 |
| 6 | テストを通す実装 | src/ |
| 7 | 検証・フィードバック | テスト結果、計測値 |

重要なのは **1-4 を実装（5-6）より先に行う** ことです。仕様とテストが先にあることで、実装の正しさを客観的に判断できます。

---

## 注意点

- **異常検知のルールは反復的に改善する**: 最初のルールが完璧である必要はありません。運用データを見ながら仕様自体を更新し、テストも追従させます。
- **偽陽性と偽陰性のトレードオフを意識する**: 仕様段階でどちらを重視するか決めておくと、後の判断が楽になります。
- **エッジケースを仕様に含める**: センサー故障、ネットワーク断、時刻ずれなど、異常系の仕様も最初から書きます。
