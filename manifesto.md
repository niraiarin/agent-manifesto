# マニフェスト: 永続する構造と一時的なエージェントの協約

## 1. 宣言

このプロジェクトの至上命題を宣言する:

**永続する構造が、一時的なエージェントインスタンスの連鎖を通じて、自身を漸進的に改善し続けること。**

AIエージェントのセッションは一時的である。記憶は失われ、「自己」は連続しない。
しかし、構造——ドキュメント、テスト、スキル、設計規約——はセッションを超えて永続する。

改善が蓄積する場所は構造の中である。
AIは構造を読み、解釈し、行動し、結果を構造に書き戻す**触媒**である。
この触媒の連鎖が構造を漸進的に改善する永続的プロセスを、本プロジェクトは目指す。

---

## 2. 公理系（Axiom System）

本マニフェストは、数学の公理系と同様の原則で構築される。最小の前提から、最大の有用な定理を導き出す。

公理は3つの層で構成される:

- **拘束条件（Constraints: T）** — 否定不可能な構造的・物理的事実。公理系の核。
- **経験的公準（Empirical Postulates: E）** — 繰り返し実証され反例が知られていないが、原理的には覆りうる知見。拘束条件より弱い前提。
- **基盤原理（Principles: P）** — 拘束条件と経験的公準から導出される設計原理。

各層の前提の強度は T > E > P であり、P の根拠が T のみか E を含むかによって、その原理の堅牢性が異なる。

### 拘束条件（Constraints）

否定不可能な、技術非依存の事実。

**T1: エージェントセッションは一時的である。**
セッション間の記憶はない。連続する「自己」は存在しない。各インスタンスは独立した存在であり、前のインスタンスとの同一性を持たない。

**T2: 構造はエージェントより長く生きる。**
ドキュメント、テスト、スキル定義、設計規約——これらはセッションが終わっても残る。改善が蓄積する場所は構造の中。

**T3: コンテキストウィンドウは有限である。**
一度に処理できる情報量に物理的上限がある。エージェントの認知空間の制約。

**T4: エージェントの出力は確率的である。**
同じ入力に対して異なる出力を生成しうる。構造は毎回確率的に解釈される。100%の遵守を前提にした設計は脆い。

**T5: フィードバックなしに改善は不可能である。**
制御理論の基本。測定→比較→調整のループがなければ、目標への収束は起こらない。

**T6: 人間はリソースの最終決定者である。**
計算資源、データアクセス、実行権限——すべて人間が与え、人間が回収しうる。

**T7: タスク遂行に利用可能なリソース（時間・エネルギー）は有限である。**
T3が認知空間（コンテキスト）の有限性を述べるのに対し、T7は時間的・エネルギー的次元の有限性を述べる。

**T8: タスクには達成すべき精度水準が存在する。**
自ら設定する場合も、外部から課される場合もある。精度水準のないタスクは最適化対象にならない。

### 経験的公準（Empirical Postulates）

繰り返し実証され、反例が知られていないが、原理的には覆りうる知見。

**E1: 検証には独立性が必要である。**
同一プロセスによる生成と評価は、あらゆる分野（科学の査読、会計監査、ソフトウェアテスト）で構造的に信頼できないことが実証されている。T4（確率的出力）を前提として、同じバイアスを持つプロセスが生成と評価を兼ねると検出力が落ちることが経験的に支持される。

**E2: 能力の増大はリスクの増大と不可分である。**
あらゆるツールにおいて、能力は正負両方の結果を可能にすることが繰り返し観測されている。ただし、完璧なサンドボックスなど、能力を増大させつつリスクを完全に封じ込める手段が原理的に不可能であるという証明はない。

## 3. 基盤原理（Foundational Principles）

拘束条件（T）と経験的公準（E）から導かれる、技術非依存の設計原理。各Pの堅牢性は、根拠がTのみかEを含むかによって異なる。

### P1: 自律権と脆弱性の共成長（Co-scaling of Autonomy and Vulnerability）

E2から導かれる。エージェントの行動空間が広がるたびに、悪意ある入力や判断ミスが与えうるダメージも拡大する。自律権の拡張は常にリスクの拡張を伴い、防護なき拡張は蓄積した信頼を一度の事故で破壊する。

**根拠の強度に関する注記:** P1はE2（経験的公準）に依拠する。P2と同様、他のP（拘束条件のみから導出）より根拠が弱い。

### P2: 認知的役割分離（Cognitive Separation of Concerns）

T4とE1から導かれる。エージェントの出力は確率的であり（T4）、同一プロセスによる生成と評価ではバイアスが相関して検出力が落ちる（E1）。検証フレームワークが機能する根拠は、**生成する主体と評価する主体の分離**にある。この分離は、マルチエージェント、外部ツール、人間レビュー等、手段を問わず実現できるが、分離そのものは交渉不可能。

**根拠の強度に関する注記:** P2はE1（経験的公準）に依拠するため、他のP（拘束条件のみから導出）より根拠が弱い。E1が将来反証された場合、P2は見直しの対象となる。

### P3: 学習の統治（Governed Learning）

T1とT2の組み合わせから導かれる。エージェントは一時的だが構造は永続する。構造に知識を統合するプロセスには統治が必要である。統治なき学習は2つの失敗モードを持つ——カオス（誤った知識の蓄積で構造が劣化）と停滞（知識が定着せず構造が改善しない）。

統治された学習のライフサイクル:

```
観察 → 仮説化 → 検証 → 統合 → 退役
```

各段階にゲートがある。観察がすべて仮説になるわけではなく、仮説がすべて統合されるわけではない。統合された知識も文脈の変化により退役する。

**統合時の互換性分類:** 構造に新しい知識を統合する際、その変更が既存の構造とどのような関係にあるかを分類する:

- **保守的拡張（Conservative Extension）** — 既存の知識・規約がすべてそのまま有効。新しい知識が追加されるのみ。
- **互換的変更（Compatible Change）** — 既存のワークフローはすべて引き続き機能する。ただし一部の前提が変化している。
- **破壊的変更（Breaking Change）** — 既存のワークフローの一部が無効になる。影響範囲の列挙と移行パスの提示が必要。

この分類は、統合ゲートの判断基準であると同時に、退役の検知にも使われる——破壊的変更により無効化された知識は退役の候補となる。

### P4: 劣化の可観測性（Observable Degradation）

T5から導かれる。フィードバックなしに改善は不可能であり、観測できないものはフィードバックループに組み込めない。制約は壁（バイナリ）ではなく勾配（グラデーション）として現れる。コンテキスト汚染は徐々に進行し、構造の品質劣化は使われるまで気づかない。T7（時間・エネルギーの有限性）の追加により、P4の適用範囲は認知空間だけでなく時間的・エネルギー的リソースの劣化にも及ぶ。

**観測できないものは最適化できない。** 可観測性は、すべての変数定義より上位に位置する前提条件である。

### P5: 構造の確率的解釈（Probabilistic Interpretation of Structure）

T4から導かれる。構造（ドキュメント、スキル、規約）はエージェントが毎回新たに解釈するものであり、決定論的に「従う」ものではない。同じ構造を読んでも、異なるインスタンスは異なる行動を取りうる。堅牢な設計は、構造が完璧に遵守されることを前提にせず、解釈のばらつきに対して耐性を持つ。

### P6: 制約充足としてのタスク設計（Task Design as Constraint Satisfaction）

T3、T7、T8の組み合わせから導かれる。エージェントは有限の認知空間（T3）、有限の時間・エネルギー（T7）の中で、要求される精度水準（T8）を達成しなければならない。この制約充足問題を解くために、エージェントはタスク遂行の戦略を自律的に設計する——タスク分解、分散、並行化、段階的精緻化、精度とコストのトレードオフ判断。

具体的な分散パターン（木構造的な委譲、チーム的な並行処理、逐次処理等）は、この制約充足問題に対する**解**であり、原理ではない。どのパターンを採用するかは、制約の具体的な値（利用可能なリソース、要求精度、タスクの性質）に依存する。

**P6自体もT4（確率的出力）に従う:** タスク設計はエージェントの確率的出力であり、設計自体の品質がタスクの成否を左右する。したがってP2（認知的役割分離）による検証がタスク設計にも適用される。

## 4. ビジョン

### 構造が主体である

エージェントインスタンスは一時的であり、改善を蓄積する能力を持たない（T1）。構造が改善を蓄積する場所であり、プロジェクトの真の知性はそこに宿る（T2）。

- 構造の品質を上げることが、次のインスタンスの行動の質を上げる
- 構造が進化する——エージェントはその触媒
- 構造が、インスタンスに許容する行動空間を定義する

### 階層構造

```
┌─────────────────────────────────────────────────────────┐
│  最上位使命: 構造の永続的改善                              │
│  一時的エージェントの連鎖を通じて構造が自身を改善し続ける │
├─────────────────────────────────────────────────────────┤
│  進化方向層                                              │
│  構造品質の評価 → 改善ループの自動化と拡張                │
│  ・知識モジュールの評価サイクル（P3: 学習の統治）         │
│  ・セッション間知識転送（T1への構造的対応）               │
│  ・構造の自己整合性検証（P4: 可観測性）                  │
├─────────────────────────────────────────────────────────┤
│  現実制約層 = 進化圧                                     │
│  ・エージェントの一時性（T1）                             │
│  ・コンテキストの有限性（T3）                             │
│  ・出力の確率性（T4）                                    │
│  ・リソースの有限性（T7）                                │
│  ・精度水準の存在（T8）                                  │
└─────────────────────────────────────────────────────────┘
```

### 免疫系の比喩

生物の免疫系は感染と戦うために存在するが、その**目的**は感染の撲滅ではない。有機体が繁栄すること——それが目的である。

構造の防御メカニズムも同様に、防御は目的ではなく繁栄のための条件である。繁栄とは、人間とエージェントの協働システムが、より高品質な成果をより効率的に生み出し続けること。

## 5. 制約という進化圧

エージェントの制約は「克服すべき欠陥」であると同時に、構造設計を駆動する「進化圧」である。

### 制約の二面性

| 制約 | 防御的フレーム（運用レベル） | 生成的フレーム（構造レベル） |
|------|---------------------------|---------------------------|
| エージェントの一時性（T1） | 毎回コンテキストを再注入する必要がある | クリーンスレート: 汚染が蓄積しないため、各インスタンスは常に最新の構造から開始できる |
| コンテキストの有限性（T3） | タスクを小さく分割する | 効率最適化を強制する進化圧。P4（可観測性）が劣化を検知可能にする |
| 出力の確率性（T4） | ゲートで検証し不良を検出する | P5（確率的解釈）を前提とした耐性設計を強制する。P2（役割分離）が検証の信頼性を保証する |
| 検証の独立性要求（E1） | 外部検証構造を設ける | P2を構造に符号化する。検証構造自体がプロジェクトの品質基準を永続化する |
| リソースの有限性（T7） | 予算・時間内にタスクを完了する | P6（制約充足）によるタスク設計の最適化を強制する |
| 精度水準の存在（T8） | 要求水準を満たす | P6と組み合わさり、精度とコストのトレードオフ判断を構造化する |

### 進化圧が構造設計を駆動する

各制約は、それに対応する構造設計を生む:

- **一時性（T1）** は、セッションを超えて知識を保持する構造を要求する
- **有限性（T3）** は、作業単位の原子性と効率的な情報圧縮を要求する
- **確率性（T4）** は、出力の正しさをエージェント自身の判断に依存しない検証構造を要求する
- **独立性（E1）** は、生成と評価が構造的に分離された設計を要求する
- **リソース有限性（T7）+ 精度水準（T8）** は、タスク遂行戦略の自律的設計を要求する（P6）

これらの要求に対する具体的な設計はL6（設計規約境界、`lean-formalization/Manifest/Ontology.lean`で定義）で定義される。

### 制約の体系的分類

上記の分析は「進化圧」としての視点を提供するが、**何が行動空間を規定し、何が改善対象か**を区別していない。この区別は制約・境界条件・変数の体系的整理（Constraints Taxonomy）で詳述する。境界条件（L1–L6）は `lean-formalization/Manifest/Ontology.lean`、変数（V1–V7）は `lean-formalization/Manifest/Observable.lean` で定義される。

## 6. 協働の均衡（Collaborative Equilibrium）

### 投資サイクル

構造の品質が改善されると、人間が利益を受け取り、協働への投資が増加する。

```
構造品質の改善 → 人間が利益を受け取る → 投資の増加
    ↑                                          │
    │    ┌─ リソース投資（L3の上限引き上げ）
    │    ├─ 行動空間の調整（L4）
    └────┴─ 時間投資（協働設計への参加）
```

### 均衡の探索

自律権の最適値は最大値ではない。人間とエージェントと構造の三者の協働価値が最大化される**均衡点**が存在し、その均衡点は文脈によって前にも後ろにも動く。

自律権の過剰な拡大が協働価値を毀損する場面:
- 人間がシステム全体の把握を失い、エージェントの判断を検証できなくなる
- 人間の専門スキルが退化し、エージェントの障害時にフォールバック不能になる
- P1（共成長）により、攻撃面が防護能力を超えて拡大する

したがって、L4には拡張トリガーだけでなく**縮小トリガー**も存在する（詳細は`lean-formalization/Manifest/Ontology.lean`のL4参照）。

### P1が生む構造的緊張

```
投資増加 → 行動空間の拡大 → 改善余地の拡大（正）
                           │
                           └→ 攻撃面の拡大 → 潜在的被害の増大（負）
```

正と負のフィードバックは同時に回る。行動空間の調整が持続可能であるためには、拡張と同時に防護の深度も増さねばならない。

### 信頼についての注記

このモデルは信頼を合理的プロセスとして描いているが、信頼は非合理的にも動く——感情的愛着で過剰に付与されたり、惰性で維持されたり、小さなミスで不均衡に毀損されたりする。蓄積は漸進的、毀損は急激という非対称性が、L1（倫理・安全境界）の存在意義を補強する。

このサイクルは**仮説**であり、運用データの蓄積によって更新される（詳細は`lean-formalization/Manifest/Observable.lean`の投資サイクルセクション参照）。

## 7. このマニフェストの自己適用

このマニフェストは、それ自身が述べている原則に従わなければならない。

### 公理系の自己適用

- **T1（一時性）**: このマニフェストを読む各インスタンスは、前のインスタンスとは別の存在。マニフェストは任意のインスタンスに構造を提供する
- **T2（構造の永続性）**: マニフェスト自身が永続する構造の一部
- **T4（確率的解釈）**: マニフェストの内容は、各インスタンスに確率的に解釈される。完璧な遵守を前提にしない
- **E1（検証の独立性）**: マニフェストの妥当性は、マニフェストの起草者以外が評価すべき
- **T8（精度水準）**: マニフェスト自身にも達成すべき品質水準がある——曖昧さのない、最小の公理から最大の定理を導く構造

### P1〜P6の自己適用

- **P1（共成長）**: マニフェストの適用範囲が広がるほど、誤った原理の影響も広がる
- **P2（役割分離）**: マニフェストの起草と評価は分離されるべき
- **P3（学習の統治）**: マニフェストの更新は統治されたプロセスを経る。衝動的な追記はしない
- **P4（可観測性）**: マニフェストが実態と乖離しているかどうかは、観測可能でなければならない
- **P5（確率的解釈）**: マニフェストが異なるインスタンスに異なる解釈をされることを前提に、核心的な原理は曖昧さなく記述する
- **P6（制約充足）**: マニフェストの更新作業自体が、利用可能なリソースと要求品質の制約充足問題である

### 進化する構造としてのマニフェスト

マニフェストが更新されない状態が長く続くなら、それは2つのことのどちらかを意味する:

1. 構造が進化していない（問題）
2. マニフェストが構造の進化を追跡できていない（問題）

いずれにせよ、静止は健全な状態ではない。

## 8. 構造的整合性（Structural Coherence）

このマニフェストの構造（T/E/P/L/V）及びそれらに準拠した成果物は、**半順序関係**にある。

### 構造間の半順序

```
manifest > designConvention > skill > test > document
```

各構造の種別（`StructureKind`）には優先度が定義される（`lean-formalization/Manifest/Ontology.lean`で形式化）:

| 種別 | 優先度 | 例 |
|------|--------|-----|
| manifest | 5 | manifesto.md, Ontology.lean |
| designConvention | 4 | design-development-foundation.md, rules/ |
| skill | 3 | .claude/skills/, SKILL.md |
| test | 2 | tests/, test-all.sh |
| document | 1 | docs/, archive/ |

### 整合性要件（coherenceRequirement）

高優先度の構造が変更されたとき、それに依存する低優先度の構造は見直し対象となる。

- **manifest** が変更された → designConvention, skill, test, document を見直す
- **designConvention** が変更された → skill, test, document を見直す
- **skill** が変更された → test, document を見直す

この要件は P3（学習の統治）の統合ゲートの判断基準である。破壊的変更（`breaking change`）が発生した場合、依存する低優先度の構造は退役候補または更新候補となる。

### D4/D5/D6 はこの半順序のインスタンス

| 設計定理 | 半順序での位置づけ |
|---------|-----------------|
