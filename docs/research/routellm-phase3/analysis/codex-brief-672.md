## 成果物 (4 ファイル + 1 Python 実装)

### 1. `scripts/router-drift-check.sh`

weekly cron/launchd wrapper。以下を実行:

1. `docs/research/routellm-phase3/classifier/` に cd
2. `uv run python3 monitor_drift.py --reference-days 7 --output ../analysis/drift-report-$(date +%Y%m%d).json` を実行
3. drift-report JSON を読んで `drift.alert == true` なら `alert_dispatcher.py --report <path>` を呼ぶ
4. stdout/stderr を `$MANIFESTO_ROOT/docs/research/routellm-phase3/logs/drift-check-$(date +%Y%m%d).log` に追記
5. exit code: drift=0 なら 0、alert=true なら 2、error=1

要件:
- **shebang**: `#!/bin/bash`
- `set -euo pipefail`
- MANIFESTO_ROOT は環境変数 or `bash .claude/skills/shared/resolve-manifesto-root.sh` で解決
- `uv` が PATH 上に無ければ `$HOME/.local/bin/uv` もフォールバック

### 3. `docs/research/routellm-phase3/classifier/alert_dispatcher.py`

drift 発生時に GitHub Issue を auto-create する Python スクリプト。

CLI:
```
uv run python3 alert_dispatcher.py --report <drift-report.json> [--dry-run]
```

必須機能:
- drift-report JSON を読み込み、`drift.alert == true` の場合のみ発火
- 同じ日の drift report で既に issue 作成済み (`docs/research/routellm-phase3/logs/drift-alert.flag` の内容と report date を比較) ならスキップ
- `gh issue create --label "router-drift" --title ... --body ...` を subprocess で呼ぶ (L1: subprocess は shell=False)
- issue 作成成功後に `drift-alert.flag` を更新 (最後のアラート日付 + report path)
- `--dry-run` 時は gh issue create を実行せず、title/body を stdout に出力
- serve_encoder.py が次回起動時に `drift-alert.flag` を読めるフォーマット (JSON 1 行)

issue title 例: `[router-drift] L1 label distribution shift 23.4pt (2026-04-30)`

issue body 例 (markdown):
```markdown
## Drift Detected

**Report**: `<path>` (generated YYYY-MM-DD)

## 指標
- label distribution L1 distance: 23.4pt (threshold: 15pt)
- confidence mean delta: -0.12 (threshold: ±0.1)
- fallback pct delta: +5.2pt
- n_total: 342 predictions across 7 days

## 推奨アクション
1. `docs/research/routellm-phase3/analysis/observability-runbook.md` §Drift Response 参照
2. 人間承認後 `uv run python3 retrain_cli.py --base-train ... --corrections ...` で再学習
3. 新 model の GT hold-out accuracy 確認 → 劣化時 rollback

## Reference
Parent: #639 / Sub-7: #672

🤖 Generated by scripts/router-drift-check.sh (launchd)
```

### 4. `serve_encoder.py` に `/metrics` endpoint 追加

**既存ファイルへの追記**。Prometheus text format で以下を export:

```
# HELP router_prediction_total Total predictions by label
# TYPE router_prediction_total counter
router_prediction_total{label="local_confident"} 123
router_prediction_total{label="local_probable"} 456
...

# HELP router_confidence_mean Mean confidence (recent 1000 predictions)
# TYPE router_confidence_mean gauge
router_confidence_mean 0.873

# HELP router_fallback_total Total fallback triggers
# TYPE router_fallback_total counter
router_fallback_total 12

# HELP router_latency_p95_ms p95 latency in milliseconds (recent 1000)
# TYPE router_latency_p95_ms gauge
router_latency_p95_ms 81.2
```

実装要件:
- `DriftLogger` クラス内に in-memory ring buffer (size=1000) を追加し、label/confidence/fallback/latency を集計
- `/metrics` endpoint は `Content-Type: text/plain; version=0.0.4` で返す
- 既存 `/classify`, `/healthz`, `/metadata` には影響を与えない

### 5. `docs/research/routellm-phase3/analysis/observability-runbook.md`

運用 runbook。目次:

1. Setup (launchd インストール手順、`sed` で `__MANIFESTO_ROOT__` 置換、`launchctl load`)
2. Verify (`launchctl list | grep router-drift`, `scripts/router-drift-check.sh` を手動実行)
3. Drift Response (drift issue を受け取ったときの対応手順: 確認 → 原因 triage → retrain 判断 → approval → 実行 → rollback 判定)
4. Teardown (`launchctl unload`)
5. Troubleshooting (gh auth 失効、uv not found、MPS OOM、log rotation 未実装の注意)
6. Metrics/Prometheus 連携 (将来展望)

## Gate 基準 (#672 Issue body から転記)

- **PASS**: 4 週間 continuous drift check 稼働 + 1 回以上の alert 発生 + 自動 rollback 成功
- **CONDITIONAL**: alert 発生したが手動対応のみ → 承認 workflow 実装
- **FAIL**: drift_report 生成失敗 or rollback 失敗

**本 PR の短期 Gate** (4 週間稼働は時間要件のため PR 単位では確認不能):
- **short-term PASS**: 全 deliverable の存在 + `scripts/router-drift-check.sh` が drift-report を生成できる + `alert_dispatcher.py --dry-run` が有効な gh issue title/body を出力 + `plutil -lint` が plist を OK とする + `curl /metrics` が Prometheus format を返す
- **short-term FAIL**: 上記いずれかが失敗

**中期 Gate** (follow-up で確認): 4 週稼働 + alert 発生時 PR 自動化

## 制約 (L1/L2/P3)

- **L1**: `gh issue create` は `subprocess.run(shell=False)` で呼ぶ (L1: command injection 回避)
- **L1**: `~/Library/LaunchAgents/` への書き込みは Codex がやらない (手動手順として runbook に記載)
- **P3**: 成果物はすべて **conservative extension**。既存 `monitor_drift.py`, `retrain_cli.py`, `serve_encoder.py` の I/O 契約は変更しない
- **D4 フェーズ順序**: L1 (安全) → P2 (検証) → P4 (可観測) → P3 (統治)。本 PR は P4 可観測性の強化
