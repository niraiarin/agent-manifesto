# 形式的導出の構成手順

## 1. 概要

### 1.1 目的

本手順書は、LLM がタスクを受け取り、Lean 4 を用いて
Γ ⊢ φ の導出を構成するまでのワークフローを定義する。

ここで:
- Γ（前提集合）: タスクの要件と前提を非論理的公理として構成した体系
- φ（目標命題）: タスクの達成条件を形式化した命題
- Γ ⊢ φ: Γ のもとで φ が導出可能であること

Lean 4 は CIC（構成的型理論）に基づくため、
存在命題 ∃x.P(x) の証明は具体的な証拠 (witness) w と P(w) の証明の構成を含む。
すなわち、導出は「達成可能である」という主張だけでなく「どのように達成するか」を内包する。

導出の構成は以下の手順で遂行する:
1. 論議領域を型として定義する
2. タスクの達成条件を目標命題 φ として定式化する
3. φ の導出に必要な前提を前提集合 Γ として構築する
4. 段階的に補題と定理を証明し、Γ ⊢ φ の導出を構成する
5. Lean コンパイラのフィードバックで修正を繰り返す
6. 最終的に Γ ⊢ φ を完成する（または Γ ⊬ φ を判定する）

### 1.2 役割分担

```
LLM（生成者）                    Lean コンパイラ（検証オラクル）
─────────────                    ──────────────────────────────
論議領域を型として定義する        型の整合性を検証する
目標命題 φ を定式化する           定義の well-formedness を検証する
前提集合 Γ を構築する             導出の正しさを検証する
Γ ⊢ φ の導出を試みる             エラーメッセージを返す
エラーを解釈し修正する            （判断しない）
戦略を選択・変更する
```

以下では特に重要な対応を記す:

## 2. Γ ⊢ φ の構成

§1.1 で定義した Γ ⊢ φ の導出を構成するために、
論議領域・目標命題 φ・前提集合 Γ の 3 要素を定義する。

### 2.1 論議領域の定義

命題が語る対象を型として定義する（用語リファレンス §3.2 論議領域）。
型定義は Γ にも φ にも属さず、両者が共有する語彙を構成する。

```lean
structure Request where
  size : Nat
  priority : Priority

structure System where
  capacity : Nat
  cacheEnabled : Bool
```

### 2.2 目標命題 φ の定式化

タスクの達成条件を命題 φ として定式化する。
達成すべき制約は φ に属する（Γ ではない）。

```lean
/-- [目標命題]
    タスク: 「このシステムは全リクエストを SLA 内で処理できる」
    形式化の意図: 任意のリクエストについて、
    システムの応答時間が SLA 上限以内であることを導出する
-/
theorem system_meets_sla :
  ∀ (req : Request) (sys : System),
    wellConfigured sys →
    responseTime sys req ≤ slaLimit :=
  sorry  -- §3 で導出を構成する
```

**φ の構成原則:**
- 一つのタスクに一つの目標命題。複合的なタスクは複数の命題に分解する
- 自然言語の意図を必ず docstring に残す（形式化ギャップ検証の基盤）
- 最初は `sorry` で全体構造を確立する。導出の構成は §3

### 2.3 Γ と φ の境界

ある命題 P がタスクから得られたとき、P を Γ に加える（仮定する）か、
φ の構成要素とする（導出すべき性質とする）かを判定する必要がある。

**判定基準:**
- P が「成立していることを前提とする事実」→ Γ に加える（前提）
- P が「成立させるべき条件」→ φ の一部とする（導出目標）

**例:**
- `slaLimit = 100` — SLA が定める定数値。事実として Γ に加える
- `∀ req, responseTime sys req ≤ slaLimit` — 達成すべき性質。φ に属する

**演繹定理（用語リファレンス §5.3）による相互変換:**
Γ, P ⊢ Q と Γ ⊢ P → Q は（多くの体系で）同値であるため、
ある命題を Γ に加えるか φ の前件とするかは形式的には等価である。
ただし、Γ に加えた命題は縮小（用語リファレンス §9.2）の対象となりうる。
φ の前件とした場合は、φ 自体の弱化（§3 Phase 3b）によってのみ変更される。

### 2.4 前提集合 Γ の構築: 基底理論と拡大

Γ を構成する非論理的公理（用語リファレンス §4.1）を、
修正ループにおける縮小（用語リファレンス §9.2）の許容範囲に基づいて
二つの部分に構成する。

```
基底理論 T₀ (Base Theory)
  外的権威に根拠を持つ非論理的公理の集合。
  修正ループにおいて T₀ に対する縮小 (contraction, §9.2) は行わない。
  ├── 契約由来 — 人間との合意、SLA、法的制約
  ├── 自然科学由来 — 物理法則、数学的事実
  └── 環境由来 — ハードウェア制約、API 仕様

  T₀ のエンコード方法:
  T₀ の主張が型定義（定義的拡大, §5.5）で表現可能な場合、
  axiom ではなく型定義 + theorem で構成する。
  列挙型の網羅性のように CIC から導出可能な性質を axiom にすると、
  公理衛生検査 2（非論理的妥当性, §2.6）に違反する。
  T₀ の権威は型の構成子の選択に反映される。
  ├── 型定義で表現可能 → 型定義 + theorem
  └── 型定義で表現不能 → axiom（公理カード必須, §2.5）

Γ = T₀ の拡大 (Extension)
  T₀ に LLM が設定する非論理的公理を追加した前提集合。
  Γ は T₀ の拡大 (§5.5) である: Thm(T₀) ⊆ Thm(Γ)
  この保存は単調性 (§2.5 / §5.3) により保証される。
  拡大部分（Γ \ T₀）に対してのみ、AGM の縮小・修正 (§9.2) を行う。
  ├── 仮説由来 — LLM が推論から設定した仮説。反証可能
  └── 設計由来 — ドメインモデルの前提。設計判断に基づく
```

**修正ループの制約:** 修正後の前提集合 Γ' について:
- T₀ ⊆ Γ'（基底理論の保存: T₀ の公理に対する縮小は禁止）
- Γ' は T₀ の無矛盾な拡大（用語リファレンス §5.5）

**T₀ への所属判定:** 公理の根拠が体系の構成者の推論・判断に依存せず、
外的権威（契約、自然法則、仕様書）に由来するものは T₀ に所属する。
構成者自身の分析・仮説・設計判断に由来するものは拡大部分に所属する。

### 2.5 公理カード

Γ の各公理には以下のメタデータを Lean の docstring として記載する。

```lean
/-- [公理カード]
    所属: T₀（契約由来）
    内容: SLA の応答時間上限は 100ms である
    根拠: SLA 契約 §3.2 に基づく
    ソース: ServiceLevelAgreement v2.1, Section 3.2
-/
axiom sla_response_time : slaLimit = 100
```

```lean
/-- [公理カード]
    所属: Γ \ T₀（仮説由来）
    内容: キャッシュヒット率は 80% 以上であると仮定する
    根拠: 過去の運用データからの推定
    ソース: LLM 推論（運用ログの統計的傾向から）
    反証条件: 実測キャッシュヒット率が 80% を下回った場合
-/
axiom cache_hit_rate_assumption :
  cacheHitRate ≥ 80
```

公理カードの必須フィールド:

| フィールド | T₀ の公理 | Γ \ T₀ の公理 | 内容 |
|-----------|----------|--------------|------|
| **所属** | 必須 | 必須 | T₀ or Γ \ T₀。根拠の種類を併記（契約/科学/環境/仮説/設計） |
| **内容** | 必須 | 必須 | 公理が何を主張するかの自然言語記述 |
| **根拠** | 必須 | 必須 | なぜこの公理を仮定するか |
| **ソース** | 必須 | 必須 | 根拠の出典（文書名、法則名、推論過程等） |
| **反証条件** | — | 必須 | どのような観察でこの公理を撤回するか（縮小の条件） |

### 2.6 公理衛生 (Axiom Hygiene)

Γ の公理の品質を保証するための 5 つの検査項目。

#### 検査 1: 非空虚性 (Non-vacuity)

公理の前件が充足可能であること（用語リファレンス §6.4）。

```
悪い例:  axiom a : ∀ (x : Empty), P x     -- Empty 型に要素がないため空虚に真
良い例:  axiom a : ∀ (x : Nat), x > 0 → P x  -- x = 1 で前件が充足可能
```

**検出方法:** 公理の前件に含まれる型が空 (uninhabited) でないか、
前件の条件が矛盾していないかを確認する。

#### 検査 2: 非論理的妥当性 (Non-logical Validity)

公理がドメイン知識なしに、純粋な論理のみで証明可能でないこと（用語リファレンス §3.2 妥当 valid）。

論理的に妥当な命題（すべての構造で真となる式）は公理にする必要がない。
それは論理法則の帰結であり、ドメイン固有の仮定ではない。

```
悪い例:  axiom a : ∀ (x : Nat), x = x       -- 反射律。論理的に妥当
良い例:  axiom a : ∀ (x : Nat), f x < f (x + 1)  -- f の性質はドメイン固有
```

**検出方法:** `by simp` や `by rfl` のみで証明が完了する場合、
その命題は公理ではなく定理として証明すべきである。

**検査 3 との違い:** 本検査は「論理法則のみで導出可能か」を問う。
検査 3 は「他のドメイン公理から導出可能か」を問う。両者は異なる。

#### 検査 3: 独立性 (Independence)

各公理が、他のドメイン公理から導出できないこと（用語リファレンス §4.3）。
独立でない公理は冗長であり、体系を必要以上に複雑にする。

```
冗長な例:
  axiom a1 : ∀ (x : Nat), x < x + 1
  axiom a2 : ∀ (x : Nat), x < x + 2   -- a1 と推移律から導出可能
```

**検出方法:** 公理 A を除去し、A が残りの公理から `theorem` として
証明可能であれば、A は冗長である。

**検査 2 との違い:** 検査 2 は「ドメイン公理なしで証明可能か」を問う。
検査 3 は「他のドメイン公理があれば証明可能か」を問う。
検査 2 に合格しても検査 3 に不合格になりうる（逆も同様）。

#### 検査 4: 最小性 (Minimality)

Γ ⊢ φ の導出に使用されていない公理が存在しないこと。

**検出方法:** 各公理を除去し、目標命題の導出が引き続き成立するかを確認する。
成立する場合、その公理は Γ にとって不要である。

#### 検査 5: 基底理論の保存 (Base Theory Preservation)

拡大部分（Γ \ T₀）の追加・修正が、基底理論 T₀ と矛盾しないこと。
すなわち、修正後の Γ' が T₀ の無矛盾な拡大（用語リファレンス §5.5）であること。

**検出方法:** 拡大部分の変更後に Lean コンパイルが通ること、
かつ T₀ の公理のみから導出される既存の定理がすべて引き続き成立すること。

## 3. Γ ⊢ φ の導出手順

§2 で定義した構造（論議領域・φ・Γ）を Lean で構築し、
Γ ⊢ φ の導出を構成する手順を定義する。

### Phase 1: Lean での構築

§2 の定義に基づき、以下の順序で Lean ファイルを構築する。

1. 論議領域を型として定義する（§2.1）
2. 目標命題 φ を `sorry` 付きで定式化する（§2.2）
3. 基底理論 T₀ を構築する（§2.4）。公理カード必須（§2.5）
4. T₀ の無矛盾性を簡易検査する
5. Γ \ T₀ の公理を追加し T₀ を拡大する（§2.4）。公理カードと反証条件必須
6. 補助定義（述語・関数・関係）を追加する

各ステップで Lean コンパイルを実行し、型エラーを解消する。
この段階でのエラーは主に型の不整合であり、
定義的拡大（用語リファレンス §5.5）で対処する。

**T₀ の無矛盾性の簡易検査について:**
公理系の無矛盾性を完全に検証することは一般に不可能である
（ゲーデルの第二不完全性定理、用語リファレンス §5.4）。
明らかな矛盾（互いに否定を主張する公理の組等）の早期検出が目的である。

### Phase 2: Γ ⊢ φ の導出の構成

**入力:** コンパイルが通る Lean ファイル（φ に sorry が残存）
**出力:** sorry が解消された Lean ファイル

#### 手順

1. **φ の分解:** 目標命題 φ の導出に必要な補題 (lemma) を同定する
2. **ボトムアップ導出:** 依存関係の少ない補題から順に導出を構成する
3. **φ の導出:** 補題を組み合わせて Γ ⊢ φ の導出を完成する

各導出の構成後に Lean コンパイルを実行する。
コンパイルが通らない場合は Phase 3（修正ループ）に入る。

### Phase 3: 修正ループ

**入力:** Lean コンパイラのエラーメッセージ
**出力:** 修正された Lean ファイル、または戦略変更の判断

#### Phase 3a: エラー解釈と修正アクション

| エラー種別 | 典型的原因 | 修正アクション | 修正の分類 |
|-----------|-----------|--------------|-----------|
| `type mismatch` | 型定義の不整合 | 論議領域の型定義を修正 | 定義的拡大 |
| `unknown identifier` | 未定義の参照 | 定義の追加 or import 追加 | 定義的拡大 |
| `tactic 'simp' failed` | simp では解けない | 別のタクティクを試行、または補題を追加 | — |
| `unsolved goals` | 導出の不足 | 追加の導出ステップ、または補題分解 | — |
| `declaration uses 'sorry'` | 未導出の命題 | sorry の解消 or Γ への公理追加の検討 | 公理追加は拡大 |

#### Phase 3b: 修正の規律

修正が体系の拡張に該当する場合、以下の分類を適用する（用語リファレンス §5.5）。

**定義的拡大（常に安全）:**
- 新しい型・関数・述語の定義追加
- 既存定義の型修正（コンパイルが通る範囲で）

**Γ \ T₀ の追加・変更（無矛盾な拡大であることを確認）:**
- 新しい公理の Γ への追加 → T₀ との矛盾チェック（検査 5: 基底理論の保存）
- 既存の拡大公理の修正 → 依存する定理の再導出
- 反証条件付きの公理カードを必ず付与

**T₀ の縮小（禁止）:**
- T₀ の公理に対する縮小が必要と判断した場合 → ワークフローを終了する（§4.3 未決）
- T₀ のどの公理が Γ ⊢ φ の導出を阻害しているかを特定する

**φ の弱化（形式化ギャップの再検証必須）:**
- 目標命題 φ や補題の条件を緩和する場合
- §2.3 の Γ / φ 境界判定を再実行する
- 弱化後の φ が元のタスクの意図をまだ捉えているか、Phase 4 で検証

#### Phase 3c: 終了判定

修正ループは以下のいずれかの条件で終了する。

**成功:**
- 全 `sorry` が解消され、Lean コンパイルが成功

**戦略変更:**
以下のパターンを検出した場合、現在の戦略を放棄して別の戦略を試みる。
- **同一エラー反復:** 同じエラーメッセージが 3 回連続で出現
- **公理膨張:** Γ \ T₀ の公理数が修正ループ開始時の 2 倍を超えた
- **複雑度増大:** 導出のステップ数が単調に増加し続ける

戦略変更の選択肢:
1. Γ \ T₀ の見直し（異なる前提を試す）
2. 論議領域の再定義（異なるモデル化を試す）
3. φ の分解方法を変更する
4. φ の弱化（最終手段。形式化ギャップの再検証必須）

**バックトラック時の保持/破棄ルール:**
- T₀（基底理論）: 常に保持。戦略変更で縮小しない
- Γ \ T₀（拡大部分）: 戦略変更時に縮小・破棄可能。新戦略で再構築する
- 論議領域の型定義: 戦略 2（再定義）の場合のみ破棄。それ以外は保持
- 導出済み補題: 依存する公理・定義が変更されない限り保持

**タイムアウト:**
- 修正試行の上限回数に到達
- 全戦略（上記 1–4）を網羅しても成功しない

**Γ ⊬ φ の判定:**
- T₀ の公理間の矛盾を発見（基底理論が無矛盾でない）
- φ の否定が導出可能（反例の構成に成功）

### Phase 4: 監査

**入力:** Γ ⊢ φ の導出が完成した Lean ファイル（sorry なし）
**出力:** 監査報告

#### 4a: 導出の完全性確認

- [ ] `lake build` で警告なしにコンパイル成功（sorry の不在を確認）
- [ ] 各定理に対して `#print axioms` を実行し、依存する公理を列挙
- [ ] Lean 基盤公理（`Classical.em`, `Classical.choice`, `propext`, `Quot.sound` 等）の使用を記録

**Lean コンパイルが保証するもの:**
`lake build` の成功は、導出項が CIC のもとで型検査に合格したことを意味する。
これは導出の形式的正しさを保証するが、
Γ の無矛盾性やモデルの存在（用語リファレンス §5.1）は保証しない。
Γ の品質は §4b の公理衛生チェックで担保する。

#### 4b: 公理衛生チェック

§2.6 の 5 検査項目をすべて実行する:
- [ ] 非空虚性: 各公理の前件が充足可能
- [ ] 非論理的妥当性: 各公理が純粋な論理のみでは導出不能
- [ ] 独立性: 各公理を除去しても他の公理から導出不能
- [ ] 最小性: 各公理を除去すると Γ ⊢ φ の導出が成立しない
- [ ] 基底理論の保存: Γ は T₀ の無矛盾な拡大

#### 4c: 形式化ギャップ検証（3 層）

**第 1 層: docstring 検査**
各公理・命題の docstring（自然言語の意図）と形式的定義を対照し、
形式化が意図を正しく捉えているかを確認する。

```
確認項目:
- φ の docstring と形式的定義の対応
- 各公理カードの「内容」フィールドと Lean の公理文の対応
- Γ / φ の境界判定（§2.3）が適切であるか
```

**第 2 層: 独立レビュー**
独立したコンテキストに以下を提示し、レビューさせる:
- 元のタスク記述（自然言語）
- Γ ⊢ φ の導出
- Γ の公理一覧（公理カード付き）

レビュー観点:
- 形式化がタスクの意図から逸脱していないか
- Γ の公理が過度に強くないか（φ を自明にしていないか）
- 導出が空虚でないか（前件が非現実的でないか）

**第 3 層: 反例探索**
意図に反するが形式的には Γ を充足するケースを探索する。

```
手順:
1. Γ をすべて満たすが、直観的にタスクが達成されないシナリオを構成する
2. そのようなシナリオが構成可能なら、形式化にギャップがある
3. ギャップが見つかった場合 → Γ の修正 or φ の修正 → Phase 1 に戻る
```

---

## 4. 終了条件

ワークフローは以下の 3 条件のいずれかで終了する。

### 4.1 Γ ⊢ φ の導出完了

Γ のもとで φ の導出が sorry なしに構成され、
Phase 4 の監査（導出の完全性・公理衛生・形式化ギャップ検証）を通過した状態。

成果物は以下で構成される:
- Γ ⊢ φ の導出（Lean ファイル）
- Γ の公理一覧（T₀ / Γ \ T₀ の分類、各公理カード付き）
- 導出の公理依存性（`#print axioms` の出力）

### 4.2 Γ ⊬ φ の判定

以下のいずれかにより、Γ のもとで φ が導出不能であると判定された状態:
- Γ ⊢ ¬φ が導出可能（φ の否定の構成的証拠）
- Γ が無矛盾でない（Γ ⊢ ⊥ が導出可能。この場合 Γ 自体の修正が必要）

### 4.3 未決

修正ループ（Phase 3c）の終了条件（タイムアウトまたは全戦略網羅）に到達し、
Γ ⊢ φ も Γ ⊢ ¬φ も構成されていない状態。
導出済みの補題と残存する sorry の一覧を報告する。
