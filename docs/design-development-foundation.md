# 設計開発基礎論: マニフェスト準拠システムの構築原理

> **位置づけ:** [マニフェスト](../manifesto.md)の公理系（T/E/P）と[制約分類](constraints-taxonomy.md)（L/V）から導かれる、**プラットフォーム非依存の設計開発原理**。
> 特定のツール（Claude Code, GitHub Actions 等）に依存せず、マニフェスト準拠システムを構築するための普遍的な基礎を定める。

---

## D1: 強制のレイヤリング原理

**根拠: P5（確率的解釈）+ L1–L6（境界条件の階層）**

マニフェストの各制約は、実装における強制力の異なるレイヤーに配置されなければならない。

### 3つの強制レイヤー

| レイヤー | 強制力 | 性質 | 配置すべき制約 |
|---------|--------|------|--------------|
| **構造的強制** | 違反が物理的に不可能 | 型システム、権限、自動ゲート | L1（倫理・安全）、L2 のうち不変なもの |
| **手続的強制** | 違反は可能だが検出・阻止される | 検証プロセス、レビュー、テスト | P2（検証）、P3（統治）、L4（行動空間） |
| **規範的指針** | 遵守は確率的（P5） | 文書、規約、指示 | P5（解釈のばらつき）、L6（設計規約） |

### 配置の原則

- **L1 に属する制約は必ず構造的強制レイヤーに配置する。** 規範的指針での L1 実装は P5 により不十分。
- **強制力が高いレイヤーほどコンテキストコスト（T3）が低い。** 構造的強制は一度設定すれば毎回読む必要がない。規範的指針は毎セッション読み込まれる。
- **境界条件のレイヤー（固定/投資可変/環境）と強制レイヤーは独立した軸。** L1（固定）が構造的強制に配置されるのは自然だが、L4（投資可変）も構造的強制に配置しうる（権限制御として）。

## D2: Worker/Verifier 分離の構造的実現原理

**根拠: E1（検証の独立性）+ P2（認知的役割分離）**

検証の信頼性は、生成と評価が**独立している**場合にのみ成立する。

### 評価検証の独立性 — 4条件

1. **コンテキスト分離**: Worker の思考過程・中間状態が Verifier に漏洩しない
2. **フレーミング非依存**: 検証基準が Worker のフレーミングに依存しない。Worker が「何を検証すべきか」を事後定義しない。Verifier は自らの基準で判断する
3. **実行の自動性**: Worker が検証を回避できない。検証の実行は Worker の裁量に依存しない
4. **評価者の独立**: Worker と異なる判断傾向を持つ主体が評価する。人間であればコンテキストを持たず十分な知識を持つ別人。LLM であれば同じコンテキストを持たない別のモデル

### リスクレベルと必要な独立性

| リスク | 対象 | 必要条件数 | 検証手段 |
|--------|------|----------|---------|
| **critical** | L1 関連（安全・倫理・権限） | 4 | 人間レビュー必須。同一モデルの Subagent では不十分 |
| **high** | 構造変更（アーキテクチャ） | 3 | Subagent（自動実行）+ 人間レビュー推奨 |
| **moderate** | 通常コード変更 | 2 | Subagent で十分 |
| **low** | ドキュメント | 1 | Subagent（手動起動でも可） |

### 分離のパターン（プラットフォーム非依存）

| パターン | コンテキスト | フレーミング | 自動性 | 評価者独立 | 充足条件数 |
|---------|-----------|-----------|--------|----------|----------|
| **人間レビュー（別人）** | ✅ | ✅ | ✅ | ✅ | 4 |
| **別モデルによる検証** | ✅ | ✅ | ✅ | ✅ | 4 |
| **自動 Subagent（同一モデル）** | ✅ | △ | ✅ | ❌ | 2–3 |
| **手動 Subagent（同一モデル）** | ✅ | △ | ❌ | ❌ | 1–2 |
| **自己レビュー** | ❌ | ❌ | ❌ | ❌ | **0 (E1 違反)** |

### 設計への含意

- **Subagent は万能ではない**: 同一モデルの Subagent はプロセスレベルの独立性のみ提供する。同じモデルが同じ種類のバグを見逃すリスクがある
- **critical リスクには人間または別モデルが必要**: L1 関連の変更を Subagent だけで検証するのは D2 の4条件に照らして不十分
- **フレーミングの排除**: Worker が Verifier に「ここを見て」と言う時点でフレーミングが入る。Verifier には成果物のみを渡し、自らの基準で検証させる

## D3: 可観測性先行の原理

**根拠: P4（劣化の可観測性）+ T5（フィードバックなしに改善なし）**

**観測できないものは改善できない。** したがって、あらゆる最適化に先立って、対象の可観測性を確立しなければならない。

### 可観測性の3条件（V 定義の前提）

各変数 V に対して:

1. **現在値が測定可能か**: 値を外部から計算する手続きが存在する（Measurable）
2. **劣化が検知可能か**: 値の悪化を、品質崩壊の前に検知できる
3. **改善が検証可能か**: 介入の前後で値の変化を比較できる

3条件のいずれかを満たさない変数は、名目上の最適化対象に過ぎず、P4 に照らして実効性がない。

### 開発順序への含意

可観測性（P4）は統治（P3）に先行しなければならない:

```
P4（何が起きているか見える） → P3（見えたものを統治する）
```

P3 の統治ライフサイクル（観察→仮説化→検証→統合→退役）の最初のステップ「観察」が P4 なしには機能しない。

## D4: 漸進的自己適用の原理

**根拠: Section 7（マニフェストの自己適用）+ P3（学習の統治）**

マニフェスト準拠システムの開発プロセス自体が、マニフェストに準拠していなければならない。ただし、初日から完全準拠は不可能なので、**各フェーズの完了時点でそのフェーズまでの準拠を達成する**。

### 自己適用の段階

| フェーズ | 完了時に達成される準拠 | 以降の開発プロセスへの影響 |
|---------|---------------------|------------------------|
| 安全基盤 (L1) | L1 準拠 | 以降、安全でない操作は構造的に阻止される |
| 検証基盤 (P2) | L1 + P2 準拠 | 以降、全ての成果物が独立検証される |
| 可観測性 (P4) | L1 + P2 + P4 準拠 | 以降、劣化が検知可能になる |
| 統治 (P3) | L1 + P2 + P4 + P3 準拠 | 以降、知識統合が統治される |
| 動的調整 | 全面準拠 | 投資サイクルが機能する |

### フェーズ順序の根拠

この順序はマニフェストの依存構造から**導出される**（任意ではない）:

1. **L1 が最初**: 他の全てに先立つ絶対制約。安全でない状態で P2 を作っても、P2 自体が危険になりうる。
2. **P2 が次**: P2 がないと、以降の実装の品質を検証する手段がない。
3. **P4 が次**: P4 がないと V が測れず、P3 の「改善されたか」が判定できない。
4. **P3 が次**: P4 で観測可能になったので、知識統合を統治できる。
5. **投資サイクルは最後**: V の測定（P4）+ 統治（P3）が揃って初めて、行動空間の動的調整が正当化される。

## D5: 仕様・テスト・実装の三層対応原理

**根拠: T8（精度水準）+ P4（可観測性）+ P6（制約充足）**

マニフェスト準拠システムの各コンポーネントは、3つの表現を持つ:

| 層 | 役割 | 例 |
|----|------|-----|
| **形式仕様** | 何が成り立つべきか（Lean axiom/theorem） | `axiom session_bounded : ...` |
| **受け入れテスト** | 実装上でそれが成り立っているか（実行可能な検証） | 「セッション終了後に状態が残らないことを確認」 |
| **実装** | プラットフォーム固有の実現手段 | Hook, Agent, Rule 等 |

### 三層の整合性

- **形式仕様 → テスト**: 各 axiom/theorem に対して、少なくとも1つの受け入れテストが存在する。テストは仕様の**操作的翻訳**。
- **テスト → 実装**: テストが先に存在し、実装がテストを通すことで準拠を証明する。
- **実装 → 形式仕様**: 実装がテストを通過しても、仕様との対応が不明確なら、仕様かテストを更新する。

### T4（確率的出力）への対応

テストは決定論的に合否を判定できるとは限らない（T4）。対策:

- **構造的テスト**: 構成の存在を確認する（Hook が登録されているか、Verifier が存在するか）。決定論的。
- **行動的テスト**: 実際に実行して結果を確認する。確率的。複数回実行して統計的に判定する。

## D6: 境界→緩和策→変数の三段設計原理

**根拠: constraints-taxonomy Part II**

設計対象を3段階に分類し、各段階で適切な設計判断を行う。

```
境界条件（不変）  →  緩和策（設計判断）  →  変数（品質指標）
```

### 設計の流れ

1. **境界条件を識別する**: プラットフォームとドメインから、動かせない制約を列挙する。
2. **緩和策を設計する**: 各境界条件に対して、その影響を軽減する構造を設計する。これは L6（設計規約）に属する判断。
3. **変数を定義する**: 緩和策の効き具合を測定する指標を定義する。P4 の3条件（測定可能、劣化検知可能、改善検証可能）を確認する。

### 逆方向の禁止

- **変数を直接改善しようとしない。** 変数は緩和策の効果の反映であり、変数自体を操作するのは Goodhart's Law の罠。
- **境界条件を変えようとしない。** 固定境界（L1, L2）は受容するのみ。投資可変境界（L3, L4）は人間の投資判断でのみ動く。

## D7: 信頼の非対称性原理

**根拠: manifesto Section 6 + P1（共成長）**

信頼の蓄積は漸進的であり上限がある。信頼の毀損は急激であり下限がない。

### 設計への含意

- **蓄積**: 行動空間の拡張は段階的に行う。一度に大きく拡張しない。各段階で V を計測し、改善を実証してから次の段階へ。
- **防御**: L1（倫理・安全）は構造的強制で実装する。規範的指針では、一度の事故で蓄積した信頼を破壊するリスクがある。
- **回復**: 毀損後の回復パスを設計に含める。行動空間の縮小（L4 の縮小トリガー）は正当な操作であり、「後退」ではなく「均衡点の再探索」。

---

## D8: 均衡探索の原理

**根拠: manifesto Section 6 + E2（能力-リスク共成長）**

最適な自律度は最大自律度ではない。人間・エージェント・構造の三者の協働価値が最大化される均衡点が存在し、均衡点は文脈によって動く。

### 設計への含意

- **行動空間は「調整」であり「拡張」ではない。** L4 は拡張トリガーと縮小トリガーの両方を持つ。
- **過剰拡張は協働価値を毀損しうる。** P1 により、行動空間の拡大はリスクの拡大と不可分。
- **均衡点は静的ではない。** 構造の品質変化、人間の状況変化、外部環境の変化により移動する。定期的な再評価が必要。

---

## D9: 分類自体のメンテナンス原理

**根拠: constraints-taxonomy Part IV**

設計基礎論自体が固定的な真実ではなく、運用の中で更新されうる。

### 自己硬直化の防止

- 各原理には「なぜこの原理か」の根拠を維持する（本文書における「根拠」欄）。根拠が失われた原理は再検討対象。
- 「D1 に書いてあるから」は正当な理由にならない。根拠（マニフェストの T/E/P/L/V）が正当化する範囲でのみ有効。
- 設計基礎論の更新は、P3 の互換性分類（保守的拡張/互換的変更/破壊的変更）に従って統治される。

---

## 原理の依存関係

```
D1（強制のレイヤリング）
  ↑ 使用: D2（P2 は手続的強制、L1 は構造的強制に配置）
  ↑ 使用: D5（テストは手続的強制の一形態）

D2（Worker/Verifier 分離）
  ↑ 使用: D4（Phase 2 で P2 を確立）

D3（可観測性先行）
  ↑ 使用: D4（Phase 3 で P4 を確立してから P3 へ）
  ↑ 使用: D6（変数定義に P4 の3条件が前提）

D4（漸進的自己適用）
  ← 使用: D1, D2, D3（フェーズ順序の根拠）

D5（仕様・テスト・実装の三層）
  ← 使用: D3（テストは可観測性の実装）

D6（三段設計）
  ← 使用: D3（変数定義の前提）
  ← 使用: D7, D8（投資サイクルの設計根拠）

D7（信頼の非対称性）
  ↑ 使用: D8（均衡探索の制約）

D8（均衡探索）
  ← 使用: D7（非対称性が均衡点の存在を示唆）

D9（メンテナンス）
  ← 適用: D1–D8 すべて（全原理が更新対象）
```
