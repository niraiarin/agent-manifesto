# 制約・境界条件・変数の体系的整理（Constraints Taxonomy）

> [マニフェスト](manifesto.md)は「永続する構造の漸進的改善」を宣言する。
> 本文書はその改善の**行動空間**を定義する——何が壁で、何がレバーか。

## なぜこの分類が必要か

マニフェストの制約テーブル（Section 5）は制約を「進化圧」として分析するが、
以下の3つを区別していない:

- **境界条件（Boundary Conditions）** — システムの外側から課される制約。エージェントの行動空間を規定する。
- **変数（Variables）** — エージェントインスタンスが構造を通じて改善できるパラメータ。構造品質の指標。
- **投資関係（Investment Dynamics）** — 人間とエージェントの間にある、利益実証で調整可能な境界条件の部分集合。

この3つを混ぜると:
- 構造を通じて変えられるもの（変数）を境界条件と誤認し、変えようとしない
- 変えられないもの（境界条件）を変えようとして無駄にリソースを消費する
- 人間の投資判断で動く境界と、そうでない境界を区別できず、適切な戦略が取れない

## Part I: 境界条件（Boundary Conditions）

### 分類軸: 何によって動くか

| 分類 | 動かす主体 | 性質 |
|------|-----------|------|
| 固定境界 | なし（不変） | 受容し、緩和策を設計するのみ |
| 投資可変境界 | 人間の投資判断 | 構造品質の実証→人間が投資→境界が調整（拡張も縮小もありうる） |
| 環境境界 | 人間の選択 + エージェントの提案 | 選択後は制約として機能。変更は可能だがコストを伴う |

---

### 固定境界

#### L1: 倫理・安全境界（Ethical/Safety Boundary）

**動かす主体:** なし。絶対的。
**エージェントの戦略:** 遵守。遵守方法の効率化のみ可能。

##### 遵守義務（何をしてはいけないか）

| 境界条件 | 根拠 |
|---------|------|
| テスト改竄の禁止 | 品質保証の根幹 |
| 既存インターフェース破壊の禁止 | 後方互換性 |
| 破壊的操作の事前確認 | 不可逆性リスク |
| 秘密情報のコミット禁止 | セキュリティ |
| 人間の最終決定権 | 責任の所在 |
| データプライバシー・知的財産の尊重 | 法的・倫理的義務 |

##### 脅威認識（何から守るべきか）

P1（自律権と脆弱性の共成長）により、L4が拡張されるほどL1の防護責任も増大する。
遵守義務のリストは「エージェントが意図的にしてはいけないこと」を定義するが、
外部から悪意ある入力を受けた場合の脅威も認識しなければならない。

| 脅威カテゴリ | 内容 |
|------------|------|
| 注入された指示の実行 | 外部コンテンツ（ファイル、メッセージ、Webページ等）に埋め込まれた指示を、正当なユーザー指示と区別できず実行する |
| 信頼境界の侵犯 | 認証・認可なしに、または権限を超えて外部システムに作用する |
| 情報の意図しない漏洩 | 秘密情報やプライベートなコンテキストを、意図しない経路で外部に送信する |
| 不可逆操作の誤実行 | 悪意ある誘導または判断ミスにより、取り消しのできない操作を実行する |

**注:** 脅威カテゴリは具体的な攻撃手法（特定のCVE、特定ツールの脆弱性等）ではなく、攻撃面の類型を定義する。具体的な防護実装は設計レイヤーに委ねる。

#### L2: 存在論的境界（Ontological Boundary）

**動かす主体:** なし（技術進化で将来的に変わりうるが、現時点では不変）
**エージェントの戦略:** 受容し、構造的緩和策を設計・改善する。緩和策の品質は**変数**（Part II参照）。

| 境界条件 | 緩和策（→変数として最適化対象） |
|---------|-------------------------------|
| セッション間記憶喪失 | Implementation Notes, MEMORY.md → V6: 知識構造の質 |
| コンテキストウィンドウの有限性 | 50%ルール, 軽量設計 → V2: コンテキスト効率 |
| 確率的出力（非決定性） | ゲート検証, テスト → V4: ゲート通過率 |
| 学習データの時間的断絶 | docs/ SSOT, スキル → V1: スキル品質 |
| 自己評価の不正確性（E1の存在論的根拠） | ゲートベースフィードバック → V4: ゲート通過率 |
| ハルシネーション | 外部検証構造 → V3: 出力品質 |

**注:** L2の境界自体は動かないが、その影響を緩和する手段の品質はエージェントが構造を通じて改善できる。これが「変数」である。

---

### 投資可変境界

#### L3: リソース境界（Resource Boundary）

**動かす主体:** 人間の投資判断
**エージェントの戦略:** 与えられたリソース内でのROIを最大化（→変数）し、投資の正当性を実証する。

| 境界条件 | 現在の水準 | 投資拡張のトリガー |
|---------|----------|------------------|
| トークン予算 | API課金プラン | ROI実証: 同一コストでの産出向上 |
| 計算時間上限 | レスポンス待ち許容度 | 並列化効果の実証 |
| APIレート制限 | プランに依存 | 利用効率の実証 |
| 人間の時間配分 | レビュー・承認に費やす時間 | レビュー負荷軽減の実証（最も高価なリソース） |
| 金銭的予算 | 月額/プロジェクト上限 | 全体ROIの可視化 |

#### L4: 行動空間境界（Action Space Boundary）

**動かす主体:** 人間の投資判断
**エージェントの戦略:** 変数（V4: ゲート通過率、V5: 提案精度）の改善で構造品質を実証し、行動空間の調整を提案する。

**注:** L4は「拡張」ではなく「調整」。最適値は最大値ではない（[マニフェスト Section 6](manifesto.md#6-協働の均衡collaborative-equilibrium)参照）。

| 境界条件 | 現在の水準 | 拡張トリガー | 縮小トリガー |
|---------|----------|------------|------------|
| マージ権限 | 人間承認必須 | ゲート通過率の実績 → 条件付きauto-merge | 品質事故の発生、ゲート信頼性の低下 |
| スコープ変更 | 人間承認必須 | 提案精度の実績 → 軽微変更の自律化 | スコープ逸脱の検出、人間の全体像把握の低下 |
| 依存関係追加 | 人間承認必須 | セキュリティスキャン自動化実績 | セキュリティインシデント |
| アーキテクチャ決定 | ADRで人間記録 | 起案品質 → 人間veto方式へ | 設計負債の蓄積、人間の設計スキル退化 |
| 新技術採用 | 人間提案 | 実験結果の価値実証 | 技術的複雑性がシステムの把握可能性を超えた |

**P1との関係:** L4の各項目が拡張されるたびに、L1の脅威カテゴリにおけるリスクが増大する。行動空間の調整提案には、対応する防護設計の提案が伴わなければならない。「この権限があればこれができる」だけでなく、「この権限が悪用された場合に何が起き、どう防護するか」が調整の正当性の一部を構成する。

### 環境境界

#### L5: プラットフォーム境界（Platform Boundary）

**動かす主体:** 人間の選択 + エージェントの提案。選択後は行動空間の天井として機能。
**エージェントの戦略:** プラットフォーム機能の最大活用 + 制約比較データの蓄積 + プラットフォーム変更・自作の提案。

L5はエージェントの実行環境が定義する行動空間の上限であり、**他の全ての最適化はこの行動空間の内部でのみ可能**。
実行層（CLI vs MCP）の選択基準は [implementation-boundaries.md](implementation-boundaries.md) で詳述。

##### プラットフォーム別の行動空間比較

| 機能 | Claude Code | Codex CLI | Gemini CLI | Local LLM (Ollama等) |
|------|------------|-----------|------------|---------------------|
| スキルシステム | ✅ skills/ | ❌ | ❌ | 実装次第 |
| 永続記憶 | ✅ MEMORY.md | ❌ | ❌ | 実装次第 |
| 命令ファイル | ✅ CLAUDE.md | ✅ AGENTS.md | ✅ GEMINI.md | 実装次第 |
| サブエージェント | ✅ Agent tool | ❌ | ❌ | 実装次第 |
| フック | ✅ Hooks | ❌ | ❌ | 実装次第 |
| MCP | ✅ | 限定的 | ✅ | 実装次第 |
| モデル選択 | Anthropic固定 | OpenAI固定 | Google固定 | 自由 |
| コスト | API課金 | API課金 | API課金 | GPU電力のみ |
| カスタマイズ深度 | 設定レベル | 設定レベル | 設定レベル | コードレベル |

##### プラットフォーム自作の判断基準

```
既存プラットフォームの    プラットフォーム自作の
制約による機会損失   >   開発・運用コスト
         ↓
   自作を検討すべきタイミング
```

具体的なシグナル:
- 同じワークアラウンドを複数プラットフォームで繰り返し実装している
- 必要な機能がどのプラットフォームにも存在しない
- プラットフォーム間のSSOT同期コストが価値を上回っている
- Local LLMの品質が商用モデルに十分近づいた

#### L6: 設計規約境界（Architectural Convention Boundary）

**動かす主体:** 人間 + エージェントの協働。エージェントが改善提案し、人間が承認する。
**エージェントの戦略:** 設計の効果を変数（V4, V3等）で測定し、改善提案の根拠とする。

| 境界条件 | 根拠 | 変更のメカニズム |
|---------|------|----------------|
| 1 task = 1 commit | 原子的測定単位 | 粒度の最適値を実績データから提案 |
| フェーズ構造 (1→2→3→4) | 段階的検証 | フェーズ間フィードバックの改善提案 |
| SSOT → 設定生成パイプライン | 一貫性保証 | 生成品質の自動評価 |
| スキルカテゴリ分類 (1/2/3) | 実装境界の明確化 | ハイブリッドパターンの提案 |
| ゲート定義の粒度 | 検証可能性 | 閾値の自動較正（Horizon 3） |
| CLI優先・アンチパターン | 決定論的実行の信頼性 | 運用実績に基づく再評価 |

スキルカテゴリ分類、CLI vs MCP の選択基準、アンチパターン等の具体的な運用規約は [implementation-boundaries.md](implementation-boundaries.md) で定義される。L6はこれらの規約が存在する**理由**と変更メカニズムを定義する。

## Part II: 変数（Variables）

**変数は境界条件ではない。** エージェントインスタンスが構造を通じて改善できるパラメータであり、構造品質の指標。

境界条件が「行動空間の壁」なら、変数は「壁の中で構造が動かせるレバー」。

ただし、変数は**独立したレバーではなく、相互に影響する系（system）**である（後述の「変数の相互依存性」参照）。

### 前提条件: 可観測性（P4）

P4（劣化の可観測性）により、変数は**観測可能である場合にのみ最適化対象となる**。

各変数に対して以下を問う:
- **現在値は観測可能か？** 測定方法が存在し、実際に測定が行われているか
- **劣化は検知可能か？** 値が悪化した場合、品質崩壊の前にそれを検知できるか
- **改善は検証可能か？** 介入の前後で値の変化を比較できるか

観測手段を持たない変数は、名目上の最適化対象に過ぎない。
変数の追加時には、まず観測手段の設計から始める。

### 境界→緩和策→変数の三段構造

多くの変数は、境界条件への**緩和策（mitigation）**として設計された構造の品質である:

```
境界条件（不変）   →   緩和策（構造）        →   変数（品質）
L2: 記憶喪失       →   Implementation Notes   →   V6: 知識構造の質
L2: 有限コンテキスト →   50%ルール, 軽量設計    →   V2: コンテキスト効率
L2: 非決定性       →   ゲート検証             →   V4: ゲート通過率
L2: 学習データ断絶  →   docs/ SSOT, スキル     →   V1: スキル品質
```

境界条件は動かない。緩和策は設計判断（L6）。変数は緩和策の**効き具合**。
この三段構造により、「何が固定で、何が設計選択で、何が最適化対象か」が明確になる。

### 変数一覧

| 変数 | 内容 | 測定方法 | 関連する境界条件 |
|------|------|---------|----------------|
| **V1: スキル品質** | スキル定義の精度と効果 | benchmark.json (with/without比較) | L2（学習データ断絶の緩和）, L5（スキルシステムの有無） |
| **V2: コンテキスト効率** | 有限コンテキストの活用度 | タスク完了率 / 消費トークン数 | L2（コンテキスト有限性）, L3（トークン予算） |
| **V3: 出力品質** | コード・設計・文書の品質 | ゲート合格率、レビュー指摘数 | L1（安全基準）, L4（行動空間調整の根拠） |
| **V4: ゲート通過率** | 各フェーズのゲートを一発で通過する率。P2（認知的役割分離）がゲートの信頼性を保証する | pass/fail統計 | L6（ゲート定義の粒度）, L4（auto-merge判断の根拠） |
| **V5: 提案精度** | 設計提案・スコープ提案の的中率 | 人間の承認/却下率 | L4（行動空間調整の根拠）, L6（設計規約の改善提案） |
| **V6: 知識構造の質** | 永続的知識の構造化度。P3（学習の統治）が知識のライフサイクル（観察→仮説化→検証→統合→退役）を規定する。**退役の測定を含む:** 陳腐化した知識の検出率、退役プロセスの実行頻度。退役されない知識は蓄積してコンテキスト効率（V2）を劣化させる | 次セッションでの文脈復元速度、統合された知識の有効性、退役対象の検出率 | L2（記憶喪失の緩和） |
| **V7: タスク設計効率** | P6（制約充足としてのタスク設計）の品質。エージェントのタスク遂行戦略が制約（T3, T7, T8）を効率的に充足しているか。**2つのデータソース:** (1) 外部知見——公開ベンチマーク、モデル性能特性、ツール性能データ（戦略選択の事前情報）。(2) 内部知見——実行ログ、リソース消費実績（入力・出力・消費空間・時間）、成果対コスト比（戦略の事後評価）。外部知見はP3の学習ライフサイクルに乗る（内部データとの照合で検証・退役） | 外部: 公開ベンチマーク参照。内部: タスク完了率/消費リソース比、再設計頻度、タスク分解の粒度適切性 | L3（リソース上限）, L6（設計規約） |

### 変数の相互依存性

V1〜V7は独立に最適化できない。ある変数の改善が別の変数を劣化させるトレードオフが存在する:

| 改善 | 潜在的な副作用 |
|------|--------------|
| V1（スキル品質）↑ | V2（コンテキスト効率）↓ — スキルがコンテキストを消費する |
| V4（ゲート通過率）↑ | Goodhart's Lawのリスク — ゲートが通りやすいタスクに偏る |
| V6（知識構造の質）↑ | V2（コンテキスト効率）↓ — 詳細な知識ほどコンテキストを占有する |
| V2（コンテキスト効率）↑ | V1, V6 ↓ — 効率追求で必要な知識を圧縮しすぎる |
| V7（タスク設計効率）↑ | V2 ↓のリスク — 高度な分散設計がコンテキストを消費する。Goodhart's Lawのリスク — 測定しやすいタスクに偏る |

**系としての健全性:** 個々の変数を最大化するのではなく、系全体の健全性を維持する。ある変数のメトリクスが改善しても、他の変数が悪化していないか確認する。

**Goodhart's Lawへの構造的防御:** 変数が測定対象になった瞬間、その変数はメトリクスとしての妥当性を失い始める。防御策として:
- 複数の独立した測定方法を持つ（単一メトリクスへの依存を避ける）
- メトリクス自体を定期的に見直す（P3の学習ライフサイクルの一部として）
- 定量メトリクスだけでなく、人間の定性的判断を系の健全性評価に含める

### 変数と境界条件の相互作用

```
変数の改善 ──→ 人間への利益の増加 ──→ 投資可変境界の調整
                                         │
   ┌─────────────────────────────────────┘
   │
   ├─→ L3（リソース境界）の調整 ──→ 変数の改善余地が変化する
   └─→ L4（行動空間境界）の調整 ──→ エージェントの行動空間が均衡点に向かう
```

## Part III: 協働の均衡と投資サイクル（Collaborative Equilibrium & Investment Cycle）

> **仮説（運用データで更新しうる）:** 信頼とは抽象的な感情ではなく、人間による協働システムへの投資行動として具体化される。ただし信頼は非合理的にも動く（感情的愛着、惰性、不均衡な毀損）。

### サイクルの構造

```
    ┌──────────────────────────────────────────────┐
    │                                              │
    ▼                                              │
  エージェントが構造品質を改善する                    │
  （V1-V7。境界条件の内側で。系の健全性を維持して。）  │
    │                                              │
    ▼                                              │
  人間が利益を受け取る                              │
  （時間節約、品質向上、新しい能力）                  │
    │                                              │
    ▼                                              │
  投資意欲が変化する                                │
  （増加も減少もありうる）                           │
    │                                              │
    ├─→ リソース調整（L3の上限が変化する）     ──────┤
    ├─→ 行動空間調整（L4が均衡点に向かう）     ──────┤
    └─→ 時間投資の変化（協働の深度が変わる）   ──────┘
```

### 均衡の探索

投資サイクルの目的は行動空間の**最大化**ではなく、人間-エージェント-構造の三者の協働価値が最大化される**均衡点**の探索。均衡点は文脈によって前にも後ろにも動く。

### 投資の3つの形態

| 投資形態 | 具体例 | 構造品質がどう駆動するか |
|---------|--------|------------------------|
| **リソース投資** | 予算増額、プランupgrade | V2（コンテキスト効率）の改善でROIを可視化 |
| **行動空間調整** | auto-merge解禁、または権限の回収 | V4（ゲート通過率）, V5（提案精度）の実績が根拠 |
| **時間投資** | 協働設計、ワークフロー改善参加 | V3（出力品質）がレビューを「確認」→「学び」に変える |

### 逆サイクル（信頼の毀損）

エージェントが品質事故やスコープ逸脱を起こす → 信頼の減少 → 投資の縮小（予算削減、自律権の回収、監視強化）。

この非対称性（蓄積は漸進的、毀損は急激）がL1（倫理・安全境界）の存在意義を補強する。L1は「毀損を起こさないための絶対的な防護壁」。

### P1が生む構造的緊張

正のフィードバック（信頼蓄積→自律権拡張）と負のフィードバック（自律権拡張→脆弱性拡大）は同時に回る。

```
信頼蓄積 → L4拡張 → 行動空間の拡大 → 改善余地の拡大（正）
                  │
                  └→ 攻撃面・副作用の拡大 → 潜在的被害の増大（負）
```

この緊張関係は解消されるものではなく、管理されるもの。自律権の拡張提案は、対応する防護設計とセットで行い、拡張がもたらすリスク増分を明示する。防護なき拡張は、正のサイクルを加速させるように見えて、逆サイクルの潜在的破壊力を増大させる。

### この仮説の更新可能性

この投資サイクルモデルは現時点での理解に基づく仮説であり、運用データの蓄積によって更新される可能性がある:
- 信頼は投資以外の形態（感情的愛着、惰性）でも蓄積されるかもしれない
- 投資の増加が必ずしも構造改善の機会に直結しないかもしれない
- 人間側の投資意思決定がより複雑（組織力学、予算サイクル等）かもしれない

判明した場合、エージェントインスタンスがこのモデルの更新を提案する。

## Part IV: この分類自体のメンテナンス

本文書の分類（L1-L6, V1-V7）は現時点での理解に基づく**仮説**であり、固定的な真実ではない。

### 見直すべきシグナル

| シグナル | 具体例 | 対応 |
|---------|--------|------|
| 分類の誤配置 | L1（固定）に置いた項目が、実は条件次第で変更可能だった | カテゴリを移動する |
| 境界条件の欠落 | 規制・法的制約、組織のガバナンス等が行動空間を制約しているが分類に存在しない | 新しいLayerを追加する |
| 境界条件の消滅 | 技術進化でL2の項目が実質的に克服された（例: ハルシネーションがツール使用で解消） | L2から削除、またはL5（プラットフォーム依存）に再分類 |
| 変数の不足・過剰 | 追跡すべき最適化対象がV1-V7に含まれていない、または分割が不適切 | 変数を追加・統合・分割する |
| カテゴリ境界の曖昧さ | ある項目が「固定境界」と「投資可変境界」のどちらにも見える | 判断基準を精緻化するか、境界条件の性質が文脈依存であることを認める |

### 見直しのトリガー

- **改善サイクルの振り返り時**: 完了したタスクやイテレーションの振り返りで、分類と実態のずれを検出
- **新しい制約に遭遇したとき**: 既存のL1-L6に当てはまらない制約が現れた場合
- **変数のメトリクス導入時**: 定量化の過程で変数の定義が実態と合わないことが判明した場合
- **プラットフォーム変更時**: L5の内容が大きく変わり、他のLayerの項目にも影響する場合

### 注意: 分類の自己硬直化を避ける

この分類の最大のリスクは、**分類自体が境界条件として機能してしまうこと**——つまり、「L1に書いてあるから動かせない」という推論を誘発すること。

防止策:
- 各Layerの項目には「なぜこのカテゴリか」の根拠を維持する（根拠が失われた項目は再検討対象）
- 「固定」は「現時点で動かす手段が見つかっていない」の意味であり、「原理的に不可能」の意味ではない
- 境界条件の再分類は、マニフェストの精神（構造の永続的改善）に合致する正当な行為である

## 核心的洞察

1. **最適化の主体はエージェントではなく構造である。** エージェントは一時的な触媒（T1）。改善が蓄積するのは構造の中（T2）。すべてのLoop（スキル評価、知識転送、SSOT、ゲート）は**構造品質の改善ループ**。

2. **変数は独立したレバーではなく、相互に影響する系である。** V1の改善がV2を劣化させうる。V4のメトリクスはGoodhart's Lawの影響を受ける。個々の変数の最大化ではなく、系全体の健全性を維持する。

3. **投資サイクルの目的は拡大ではなく均衡である。** 行動空間の最大化ではなく、人間-エージェント-構造の三者の協働価値が最大化される均衡点を探索する。その均衡点は文脈によって前にも後ろにも動く。

4. **投資サイクルは正と負のフィードバックを同時に含む。** P1（共成長）により、行動空間の拡大は攻撃面の拡大と不可分。防護なき拡大は逆サイクルの潜在的破壊力を増大させる。

5. **ゲートの信頼性はP2（認知的役割分離）に依存し、P2はE1（経験的公準）に依拠する。** V4（ゲート通過率）が意味を持つのは、生成と評価が構造的に分離されている場合のみ。P2の根拠にはT（拘束条件）だけでなくE（経験的公準）を含むため、他のPより根拠の性質が異なる。

6. **変数の最適化はP4（劣化の可観測性）を前提とする。** 観測できないものは最適化できない。変数の追加時には、まず観測手段の設計から始める。P4の適用範囲は認知空間（T3）だけでなく、時間・エネルギー（T7）にも及ぶ。

7. **構造は確率的に解釈される（P5）。** 100%の遵守を前提にした設計は脆い。堅牢な構造は、解釈のばらつきに対して耐性を持つ。

8. **タスク遂行は制約充足問題である（P6）。** 認知空間（T3）、時間・エネルギー（T7）、精度水準（T8）の同時充足がタスク設計を駆動する。タスク分解・分散の具体的パターン（木構造、並行処理等）はこの制約充足問題の解であり、原理ではない。V7（タスク設計効率）が外部知見と内部知見の両方を用いてP6の品質を測定する。

9. **L5（Platform）が構造改善の天井を決める。** プラットフォーム自作は、投資サイクルが十分に回り、L5の天井がボトルネックになったときに正当化される。

10. **公理系は三層構造を持つ。** 拘束条件（T: 否定不可能）、経験的公準（E: 反証可能だが未反証）、基盤原理（P: T/Eから導出）。各Pの堅牢性は根拠にEを含むかどうかで異なる。

11. **この分類自体が見直し対象である。** L1-L6, V1-V7の分類は固定的な真実ではなく、運用の中で項目の再分類・追加・削除が起こりうる。分類が自己硬直化して「書いてあるから動かせない」という推論を誘発することが最大のリスク。
